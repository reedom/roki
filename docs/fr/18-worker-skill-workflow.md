# FR 18: Phase Subprocess Catalog

> The catalog of bounded phase subprocesses the orchestrator session nominates per ticket. The phase set and per-phase defaults are mode-aware: `SPEC_DRIVEN` mode (operator declared a project-level spec is ready via the `roki:impl` label) drives the kiro spec pipeline; `NEEDS_CLASSIFY` mode (operator authorized via `roki:ready` only) starts with a `classify` phase that branches between auto-implement (Path B) and operator handoff (Path A / C / D / E). Each phase ships a daemon-built default invocation (a slash-command-driven skill or a daemon-internal prompt template); operators MAY override per phase via `WORKFLOW.md`. Documents the per-phase exit envelope the orchestrator reads, and the artifacts produced inside selected phases (`review.md` from `finalize_review`) that the orchestrator then validates structurally per [FR 19 §Artifact validation](19-orchestrator-session.md).

## Purpose

[FR 19: Orchestrator Session](19-orchestrator-session.md) describes the long-lived "thinking" component that absorbs target-spec resolution (SPEC_DRIVEN), classify-driven path branching (NEEDS_CLASSIFY), phase planning, structural artifact validation, daemon-only failure-directive processing, and Linear writes via Linear MCP. The orchestrator does not edit code; the actual code-changing work runs in **short-lived bounded phase subprocesses** that the orchestrator nominates via `action=run_phase`. [FR 07: Worker Execution](07-worker-execution.md) describes the engine adapter's mechanical supervision of those subprocesses (launch flags, stall detection, stream-json parsing, exit translation into `phase_complete` / `phase_nonclean` events).

This FR canonicalizes the **per-phase catalog**: which default skill (or daemon-internal prompt template) drives each phase in each mode, what override surface operators expose to swap or replace that default, what the daemon expects in the structured exit envelope, and what the orchestrator reads from the resulting `phase_complete` / `phase_nonclean` event to decide its next action. The previous per-issue `materialize_spec` phase is removed — the operator's choice between SPEC_DRIVEN and NEEDS_CLASSIFY (expressed via the `roki:impl` Linear label, per [FR 04 §Pre-admission judge](04-state-machine-and-recovery.md)) replaces it. SPEC_DRIVEN reuses an already-completed project-level spec (`<repo>/.kiro/specs/<target>/`) authored by the operator outside the daemon flow; NEEDS_CLASSIFY runs `roki-classify` to decide whether the work is small enough for direct implementation against the ticket body alone (Path B) or whether the operator must take it back for spec authoring (Path A / C / D / E).

## User-visible Behavior

### Phase catalog

The orchestrator's `action=run_phase` directive carries a `phase` value drawn from the catalog below. For each phase the daemon spawns a single subprocess inside the issue's session tempdir, with its own `--max-turns` budget and the per-phase context envelope (including the orchestrator's `additional_context` verbatim through the engine adapter's `additional_context` channel — see `req:roki-mvp:13`). The invocation form follows the phase override surface (see §Phase override): the catalog default for each phase is either a slash-command-driven skill (`claude -p '/<skill> <args>' --output-format stream-json --max-turns N`) or a daemon-internal templated prompt rendered onto stdin (`claude --input-format stream-json --output-format stream-json --max-turns N`).

Phase legality is mode-dependent. `classify` is only legal in NEEDS_CLASSIFY mode and only as the first nominated phase. `implement` / `review` / `validate` use different defaults per mode. `open_pr` / `ci_fix` / `finalize_review` are mode-agnostic.

Slash commands work as the initial prompt argument in `-p` mode (the prompt string is parsed before headless takes over), so a SKILL.md `disable-model-invocation: true` flag (e.g. on `kiro-impl` or `roki-ci-fix`) does not prevent the daemon from launching that skill via `/<skill> <args>`. Authoring-time skills (`kiro-spec-init`, `kiro-spec-requirements`, `kiro-spec-design`, `kiro-spec-tasks`, `kiro-spec-batch`, `kiro-spec-quick`, `kiro-spec-status`, `kiro-validate-design`, `kiro-validate-gap`, `kiro-discovery`, `kiro-steering`, `kiro-steering-custom`) are operator-side only and are not invoked by the daemon. `roki-classify` reuses the lightweight-scan + path-determination logic of `kiro-discovery` but is a separate skill: it has no dialogue steps and emits a structured exit envelope the orchestrator can branch on.

A Type B (with-human-planning) ticket is handled inside an `implement` phase: the phase agent uses the operator-installed Linear MCP (per [FR 11: Agent Tool Boundary](11-agent-tool-boundary.md)) to post questions as ticket comments and poll for replies. There is no dedicated `kiro-plan-with-human` skill today — this is normal Claude Code session work using Linear MCP and Bash, scoped to the same `implement` phase budget.

### Phase override (operator)

Each phase ships a daemon-built default invocation (catalog above). Operators MAY override per phase via the workspace-level `WORKFLOW.md` ([02-configuration](02-configuration.md)) using either of two mutually exclusive forms:

1. **Slash-command override** — `extension.phase.<name>.command = "/<custom-skill> <args...>"` swaps in a different slash-command-driven skill while keeping the daemon's invocation pattern (`claude -p '<command>' --output-format stream-json --max-turns N`). Variable substitution against the same per-phase context envelope (issue id, target spec name, repo, mode, etc.) is applied to the command string before launch.
2. **Template override** — a `prompt_template_<name>` named template block (Liquid + Markdown) replaces the default prompt entirely; the daemon renders the block against the same per-phase variables, writes the rendered text to the subprocess's stdin, and launches `claude --input-format stream-json --output-format stream-json --max-turns N`.

The two forms are mutually exclusive per phase: declaring both for the same phase is a configuration error and is rejected at startup or retained as the previous policy at hot reload (per `roki-mvp Req 6.7`). When neither form is declared the daemon uses the catalog default for the active mode.

For phases whose default is itself a daemon-internal template (`implement` direct mode, `validate` direct mode, `open_pr`), the override forms are identical to all other phases; the only difference is that the catalog default is `prompt_template_<name>` rather than a slash-command skill.

Override applies per ticket admission of that phase: an in-flight phase subprocess always finishes with the configuration that was in effect when the daemon spawned it; subsequent nominations of the same phase pick up the new policy.

The phase subprocess's tool surface, sandbox profile, and `--max-turns` budget are governed by [07-worker-execution](07-worker-execution.md) and the operator's permission strategy regardless of which override form is used. An override changes the prompt or the slash command — it does not change the daemon-side supervision contract.

The `kiro-*` skills referenced as catalog defaults provide reusable spec-driven dev protocol (implementation, code-quality review, acceptance validation, debug). The `roki-*` skills provide roki-specific lifecycle protocol (classify, CI-failure triage, `review.md` synthesis) for which kiro has no equivalent. Operators can swap either with their own slash-command-driven skill or templated prompt by declaring the override.

Operators may swap any default via the phase override surface (above).

### Per-phase exit envelope (phase subprocess → daemon → orchestrator)

The phase subprocess emits a terminal stream-json `result` event on clean exit. The daemon does not interpret reasoning text (per [FR 07: Worker Execution](07-worker-execution.md) §Boundaries); it observes the structured `result` and translates the outcome into one of two events delivered to the orchestrator's stdin:

- **`phase_complete`** — clean exit with `result.subtype = success`. Payload includes `phase` (which one just ran), the parsed `result` envelope, the `pr_url` when the phase produced one (currently `open_pr`), the `review_artifact_path` when the phase wrote one (`finalize_review`), and any phase-specific summary fields the skill emitted in its `result` payload. For `classify` the payload additionally carries `result.path ∈ {A, B, C, D, E}`, `result.suggested_command`, and `result.suggested_label`. The orchestrator reads this and returns either `action=run_phase` (next phase) or `action=stop` (`outcome=success | failure | needs_operator | ...`).
- **`phase_nonclean`** — non-zero exit, signal termination, stall-detected termination, exhausted `--max-turns`, or terminal `result` with a non-`success` subtype (including a subtype the daemon's compiled mapping does not recognize, in which case the raw subtype is forwarded to the orchestrator — see [FR 07](07-worker-execution.md) §Termination handling). Payload includes the failure classification, the raw subtype when applicable, and the verbatim phase-`additional_context` the orchestrator had passed in. The orchestrator reads this and decides whether to re-run the same phase, fall through to a `ci_fix` phase, or `action=stop`.

The standalone `ROKI_RESULT:` final-line contract from the prior single-worker model is **removed**. Each phase emits its own structured exit through the stream-json `result` event, and the daemon maps the `result` into the `phase_complete` / `phase_nonclean` events the orchestrator consumes. The orchestrator's strict JSON action contract ([FR 19 §Response schema](19-orchestrator-session.md)) is the only "thinking" surface the daemon parses; intra-phase advisory progress markers stay in the structured log but are not state-machine inputs.

### Acceptance criteria convention (EARS)

Every Linear ticket eligible for roki dispatch MUST include an `## Acceptance Criteria` section in its body, written in EARS style — preferably the Ubiquitous (`The X shall Y.`) and Event-driven (`When <trigger>, the X shall Y.`) patterns, with each criterion numbered. In NEEDS_CLASSIFY (Path B / direct) mode this section is the sole authoritative criteria source consumed by `implement` / `review` / `validate` / `finalize_review`. In SPEC_DRIVEN mode the project-level `requirements.md` is the criteria source, but the ticket body's `## Acceptance Criteria` should still describe the slice this PR delivers (informational; the orchestrator does not cross-check the two).

The daemon's pre-admission-judge does not enforce the convention (it is a content rule, not a label rule, per [FR 04 §Pre-admission judge](04-state-machine-and-recovery.md)). A malformed ticket body falls through to a downstream check inside the first phase that needs a numbered criterion list — typically `implement` — and surfaces as a `phase_nonclean` the orchestrator can route to operator handoff.

### Produced artifacts

One phase writes an artifact at a canonical path under `<workspace_root>/<repo>/<issue>/.kiro/specs/<issue>/` ([ref:artifacts](../reference/artifacts.md)). It is not written under the session tempdir.

- `finalize_review` writes `review.md`. After clean exit the orchestrator reads it and validates structurally per [FR 19 §Artifact validation](19-orchestrator-session.md).

The schema is the canonical one defined in [ref:artifacts](../reference/artifacts.md): an overall `status` of `pass | fail`, per-criterion entries indexed by the criterion ID source for the active mode (SPEC_DRIVEN: numeric requirement IDs in `<repo>/.kiro/specs/<target>/requirements.md`; direct mode: numbered EARS sentences in the ticket body), and `code_references` (an array of workspace-relative file paths with optional line range) on each entry whose status is `pass`. This FR does not introduce a new schema; it only specifies which phase produces the artifact and how the criterion ID source is selected.

### Cancellation / timeout

If the daemon SIGTERMs a phase subprocess (per [FR 07: Worker Execution](07-worker-execution.md) stall handling), the active skill SHOULD treat it as cancellation: stop dispatching new subagents, allow the current subagent to wind down briefly, and exit. The worktree and session tempdir are preserved so a human can inspect intermediate state. The daemon then sends `phase_nonclean` to the orchestrator with the stall classification; the orchestrator decides whether to retry, fall through to `ci_fix`, or `action=stop`.

## Capabilities

- **Per-phase isolation**: each phase = its own bounded subprocess with its own `--max-turns` budget, its own driving prompt (default skill, default daemon-internal template, or operator override), and its own structured exit envelope. The daemon supervises lifecycle uniformly per [FR 07](07-worker-execution.md).
- **Mode-aware defaults**: SPEC_DRIVEN uses the kiro spec pipeline (`kiro-impl` / `kiro-validate-impl` / `kiro-review`) against an operator-completed project-level spec. NEEDS_CLASSIFY starts with `roki-classify`, then either continues with daemon-internal direct-mode templates (Path B) or hands off to the operator (Path A / C / D / E) — no per-issue spec materialization.
- **Orchestrator nominates, daemon spawns**: the orchestrator is the only "thinking" component that decides which phase runs next; the daemon never selects a phase on its own. The retry decision on a `phase_nonclean` and the `review.md` validation retry decision after a clean `finalize_review` exit are both the orchestrator's, not the daemon's.
- **Defaults + per-phase override**: each phase ships a daemon-built default (mode-aware where it matters); operators replace any phase's invocation via `extension.phase.<name>.command` (slash-command swap) or `prompt_template_<name>` block (templated stdin). The override surface is uniform across the catalog.
- **Composes kiro skills + roki-specific skills**: kiro skills cover reusable spec-driven dev protocol (`kiro-impl` / `kiro-review` / `kiro-validate-impl` / `kiro-debug` / `kiro-verify-completion`). roki-specific skills cover roki lifecycle that kiro does not address (`roki-classify` / `roki-ci-fix` / `roki-finalize-review`).
- **Slash-command headless invocation**: `/<skill> <args>` works as the initial prompt argument in `-p` mode, even for skills whose manifest sets `disable-model-invocation: true` (the slash-command entry path is unaffected by the model-invocation flag).
- **Operator-installed tools, no daemon proxy**: every Bash / `gh` / Linear-MCP call from inside a phase goes through the operator's Claude Code tool surface unchanged ([FR 11: Agent Tool Boundary](11-agent-tool-boundary.md)).
- **Single artifact validated by the orchestrator**: `finalize_review` writes `review.md`; the orchestrator reads it after the producing phase clean-exits and validates it structurally per [FR 19 §Artifact validation](19-orchestrator-session.md). No daemon-side gate hook subscribes.

## Boundaries

- **The daemon does not drive these phases** — it spawns the phase subprocess the orchestrator nominated, observes its lifecycle, and forwards the structured exit to the orchestrator. State-machine transitions come from subprocess exit and Linear state, not from any phase-internal event ([FR 07: Worker Execution](07-worker-execution.md)).
- **The daemon does not enforce the phase order** — the orchestrator's `action=run_phase` directives are the only sequencing signal. The orchestrator's `max_phases` budget is the daemon-side cap; per-phase order is the orchestrator's choice. The only structural constraint is that `classify` is legal only as the first phase in NEEDS_CLASSIFY mode.
- **Loop budgets are skill-internal** — `kiro-impl`'s per-task review loop, `roki-ci-fix`'s per-category fix loop, and `kiro-validate-impl`'s mechanical / spec staging all live in skill state, not in any daemon-side persistent column. Cross-phase budgets (ticket-level retry, `max_phases`) live on the orchestrator and on the daemon, not in the skill.
- **Skill internals are skill-owned** — the per-task review loop, validation-command discovery, CI log categorization, and remediation strategies live inside `kiro-impl` / `kiro-validate-impl` / `kiro-review` / `kiro-debug` / `roki-ci-fix` / `roki-finalize-review` and are not re-specified here. This FR only specifies how the orchestrator composes them through the phase catalog.
- **No daemon-side per-issue spec materialization** — the previous `materialize_spec` phase is removed. SPEC_DRIVEN reuses an existing project-level spec the operator authored outside the daemon flow; NEEDS_CLASSIFY's `classify` phase decides whether the work needs a spec at all and (when it does) hands off to the operator. The daemon does not invoke `kiro-discovery` / `kiro-spec-init` / `kiro-spec-requirements` / `kiro-spec-design` / `kiro-spec-tasks` / `kiro-spec-quick` / `kiro-spec-batch` — these remain operator-side authoring tools.
- **No standalone `ROKI_RESULT:` line** — superseded by the per-phase stream-json `result` event mapped into `phase_complete` / `phase_nonclean` events.
- **No daemon-driven retry of the inner phases** — retries inside a phase are skill-internal; the daemon-level retry budget ([FR 07: Worker Execution](07-worker-execution.md)) only frames how many `phase_nonclean → run_phase` cycles the orchestrator may drive before `daemon_directive (kind=retry_exhausted)`. `review.md` validation retries after `finalize_review` clean exit are orchestrator-driven (see [FR 19 §Artifact validation](19-orchestrator-session.md)) and re-use the same `action=run_phase` channel; they do not consume a phase-`nonclean` retry slot because the producing phase exited cleanly. Likewise, `review` REJECTED → `implement` re-nomination and `validate` NO_GO → `implement` re-nomination are orchestrator-driven and counted against `max_phases` (and the ticket-level retry budget if the producing phase exited non-clean).
- **Phase override does not change the daemon contract** — `extension.phase.<name>.command` and `prompt_template_<name>` swap the prompt or skill body but not the lifecycle observation, exit-envelope translation, or per-phase tool surface; an override that needs more tools must declare them on the operator's Claude Code allowlist independently.
- **Auto-merge is out of scope here** — `open_pr` ends at "PR opened". Whether the daemon then auto-merges on CI green is governed elsewhere (currently out of MVP).
- **Multi-engine portability is out of scope** — the contract assumes Claude Code stream-json. If non-Claude engines are added later, the skill set abstraction (kiro + roki) needs re-expression in those engines (deferred).
- **Mode mutation mid-flight is out of scope** — see [FR 04 §Pre-admission judge](04-state-machine-and-recovery.md). Relabeling a ticket does not re-route an in-flight orchestrator.
