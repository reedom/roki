# Doc Kinds Manifest

Machine-readable kind definitions consumed by `roki-doctools`. Path globs, ID patterns, and INDEX output locations live here, not in source code.

The CLI loads `${ROKI_DOC_ROOT}/kinds.md` at startup (default `ROKI_DOC_ROOT=docs`). The single fenced YAML block below is the source of truth — prose around it is for humans only.

For schema semantics (what `implements` / `depends_on` / `related` / `provides` / `modules` mean), see [`.kiro/steering/refs.md`](../.kiro/steering/refs.md).

## Adding a new kind

1. Add an entry to the YAML block below.
2. (Optional) Add `index.output` if the kind should have a generated INDEX file.
3. Run `roki-doctools validate` to confirm no existing docs break.
4. Frontmatter on existing docs that use the new kind is yours to add.

The CLI accepts any kind name listed here; no code change needed.

## Manifest

```yaml
# Each entry is a kind definition.
#   name           — kind identifier used as the `kind:` field in front matter
#                    and as the prefix in `id:` (e.g., kind=fr → id=fr:01-...).
#   path_globs     — repo-relative globs the scanner walks. Omit for kinds
#                    declared only via `provides:` of another doc.
#   declared_via   — "provides" if this kind never has its own file
#                    (the IDs are listed in the `provides:` of another doc).
#   id_pattern     — informational; what an ID for this kind looks like.
#                    Not enforced by the validator.
#   singleton      — true if there is exactly one doc of this kind
#                    (e.g., roadmap).
#   index          — when set, `roki-doctools index` writes a generated
#                    INDEX for this kind to index.output. The generated
#                    file carries frontmatter (kind=index, generated=true,
#                    id=index:<name>).
kinds:
  - name: roadmap
    path_globs: [".kiro/steering/roadmap.md"]
    singleton: true
    id_pattern: "roadmap"

  - name: brief
    path_globs: [".kiro/specs/*/brief.md"]
    id_pattern: "brief:{spec}"

  - name: requirements
    path_globs: [".kiro/specs/*/requirements.md"]
    id_pattern: "requirements:{spec}"

  - name: req
    declared_via: provides
    id_pattern: "req:{spec}:{n}"

  - name: design
    path_globs:
      - ".kiro/specs/*/design.md"
      - ".kiro/specs/*/design-*.md"
    id_pattern: "design:{spec}[:slug]"

  - name: tasks
    path_globs: [".kiro/specs/*/tasks.md"]
    id_pattern: "tasks:{spec}"

  - name: research
    path_globs: [".kiro/specs/*/research.md"]
    id_pattern: "research:{spec}"

  - name: fr
    path_globs: ["docs/fr/[0-9]*.md"]
    id_pattern: "fr:{slug}"
    index:
      output: docs/fr/index.md

  - name: reference
    # `[a-z]*` excludes README.md (uppercase) by design; index.md is included
    # because it carries its own kind: index frontmatter.
    path_globs: ["docs/reference/[a-z]*.md"]
    id_pattern: "ref:{slug}"
    index:
      output: docs/reference/index.md

  - name: example
    # docs/examples/*.md files (WORKFLOW.{minimal,annotated}.md) carry
    # application-level frontmatter, not refs:. Path glob points at a
    # non-matching pattern so the strict validator does not require them.
    # Add a real example doc with a refs: block to enable indexing later.
    path_globs: ["docs/examples/*.example.md"]
    id_pattern: "example:{slug}"

  - name: crate
    path_globs: ["crates/*/README.md"]
    id_pattern: "crate:{name}"
    index:
      output: crates/index.md

  - name: index
    declared_via: generated
    id_pattern: "index:{kind}"
```

## Reserved kind: `index`

Files generated by `roki-doctools index` carry `kind: index` and are never hand-edited. Their `provides:` enumerates the IDs they index over so the graph treats them as first-class nodes.

## Out of scope

- No glob expansion in the `path_globs` beyond standard shell globbing.
- The `id_pattern` field is documentation, not enforcement; ID format is left to authors and validated by uniqueness only.
- Kinds defined here that have no path_globs and no `declared_via` will fail validation.
