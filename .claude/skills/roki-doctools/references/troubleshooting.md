# roki-doctools — Troubleshooting

Each entry: error message → root cause → fix.

## `<file> (<id>): dangling reference \`<target>\``

**Cause.** A doc cites `<target>` in its `implements` / `depends_on` / `related` list, but no doc declares it (neither as a primary `id:` nor in any `provides:` list).

**Likely fixes:**
- The producer forgot to list the ID in `provides:`. Edit the producer doc and add the ID to its `provides:` list.
- The citation is a typo — fix it in the offending file.
- The producer was renamed and not all citations were updated — search-and-replace.
- The producer was deleted — remove the citation, or recreate the producer.

## `<file> (<id>): dangling reference \`<target>\` (related)`

Same as above, but for a `related:` entry. Soft edges are still validated.

## `duplicate id \`<id>\` in <file-A> and <file-B>`

**Cause.** Two `refs:` blocks declare the same primary `id:`.

**Fix.** Decide which file owns the ID. The other must change its `id:` (or be deleted, if it's redundant).

## `duplicate id \`<id>\` provided by both <file-A> and <file-B>`

**Cause.** Two `provides:` lists declare overlapping IDs (or one provides an ID that another file uses as its primary `id:`).

**Fix.** Same — pick one owner, remove from the other.

## `<file> (<id>): unknown kind \`<kind>\` (not declared in kinds.md)`

**Cause.** The frontmatter's `kind:` value doesn't appear in `docs/kinds.md`.

**Fix:**
- Typo in the doc → correct it.
- The kind genuinely is new → add it to the YAML block in `docs/kinds.md` (name + `path_globs` + `id_pattern`; add `index.output` if a generated index is wanted).

## `<file> (<id>): module path \`<path>\` does not exist`

**Cause.** A path in `modules:` was renamed or deleted on disk.

**Fix.** Update the `modules:` entry to the new path, or remove it if the doc no longer documents that file.

For directory prefixes (trailing slash), the directory must exist; an empty directory is fine.

## `<file>: matches kind \`<kind>\` glob \`<glob>\` but has no \`refs:\` block`

**Cause.** A file matched by a kind's `path_globs` has no `refs:` frontmatter. Strict glob coverage requires every matched file to be in the graph.

**Fix:**
- The file should be in the graph → add a `refs:` block, mirroring the closest sibling.
- The file should NOT be in the graph (e.g., README, template, scratch, generated by external tooling) → tighten the kind's `path_globs` in `docs/kinds.md` to exclude it.

The reference manifest already excludes `README.md` from `docs/reference/[a-z]*.md` by case (lowercase initial). Use the same pattern for new exclusions when applicable.

## `<file>: front matter parse error: <yaml-error>`

**Cause.** YAML syntax error inside the `---` fences.

**Fix.** Common culprits:
- Unquoted strings containing `:` or `#`.
- Inconsistent indentation (must be spaces, not tabs).
- A list item that should be `- foo` written as `- "foo"` mixed with bare strings causing parser confusion.

Open the offending file and re-validate the YAML by eye. If unsure, paste into a YAML linter.

## `kind \`<name>\`: bad glob \`<pat>\`: <error>`

**Cause.** A `path_globs` entry in `docs/kinds.md` is not a valid glob.

**Fix.** Check the glob syntax against the standard `glob` crate rules: `*`, `**`, `?`, `[...]`. No brace expansion; no glob negation.

## `read kind manifest at <path>: ...`

**Cause.** `${ROKI_DOC_ROOT}/kinds.md` is missing or unreadable.

**Fix.** Confirm `ROKI_DOC_ROOT` (default `docs`) is correct, that the working directory is the repo root (or `--root` is set), and that `docs/kinds.md` exists.

## `parse kind manifest at <path>: ...`

**Cause.** `docs/kinds.md` exists but the fenced YAML block does not parse.

**Fix.** Open `docs/kinds.md`, locate the ```` ```yaml ```` ... ```` ``` ```` block, and fix the YAML syntax. The CLI requires exactly one such fenced block.

## `at least one id is required` / `unknown id(s): <list>`

**Cause.** `impact` / `deps` / `show` / `touched` was invoked with no arguments or with an unknown ID.

**Fix.** Use `roki-doctools list | grep <fragment>` to discover the right ID, then re-run.

## Behavior surprises

### "I added a new fr but `index` didn't pick it up."

Run `cargo run -q -p roki-doctools -- index` after the edit. The hook does not regenerate indexes — it only runs `validate` and `touched`.

### "validate passes but the prose `## Traceability` section is wrong."

The validator does not parse prose. Trust front matter as the source of truth and update the prose section to match.

### "`touched <path>` returns no docs of record."

No doc claims that path in `modules:`. If the file is public surface (a CLI flag, a config key, an extension point, etc.), pick the relevant fr or reference doc and add the path to its `modules:`.

### "The hook is reporting an error I already fixed."

The hook runs on every save. After fixing, save the file again (or save any other `.md` file in the same edit cycle) and the hook re-runs.

## Diagnostic playbook

When facing an unfamiliar error:

1. Read the error message — it names the offending file and ID.
2. `cargo run -q -p roki-doctools -- show <id>` to inspect the doc's frontmatter and immediate edges.
3. `cargo run -q -p roki-doctools -- list | grep <fragment>` to confirm a candidate ID exists.
4. Re-read `docs/reference/frontmatter.md` and `docs/kinds.md` to confirm the schema rule that's being violated.
5. If the rule is unclear, check the validator source at `crates/roki-doctools/src/main.rs` (functions `cmd_validate`, `build_edges`, `module_covers`).
