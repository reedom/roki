# `refs:` Frontmatter Cheatsheet

Authoritative source: `docs/reference/frontmatter.md` (schema) and `docs/kinds.md` (kind manifest). This file is a quick lookup; consult the originals when behavior is ambiguous.

## Block shape

```yaml
---
refs:
  id: <kind>:<scope>[:<sub>]   # required, globally unique
  kind: <kind>                  # required, must match a name in docs/kinds.md
  title: "<free text>"          # optional
  spec: <spec-name>              # optional, parent spec; null for cross-spec docs
  provides:                      # optional, IDs declared inside this file
    - <id>
  implements:                    # optional, IDs this file fulfills (hard upstream)
    - <id>
  depends_on:                    # optional, hard upstream IDs
    - <id>
  related:                       # optional, soft see-also (bidirectional)
    - <id>
  modules:                       # optional, repo-relative source paths or dir prefixes
    - <path>
    - <path-prefix>/             # trailing slash means directory prefix
---
```

All list fields default to empty.

## Field semantics

| Field | Direction | Strength | Meaning |
|---|---|---|---|
| `implements` | downstream → upstream | hard | "I exist to satisfy this upstream artifact." |
| `depends_on` | downstream → upstream | hard | "I would be incorrect or incomplete without this." |
| `related` | bidirectional | soft | "See also." Folded into traversal only with `--include-related`. |
| `provides` | self → child IDs | hard | "I declare these additional IDs inside my body." |
| `modules` | doc → code | hard | "I am the design of record for these source paths." |

The forward graph (`implements` ∪ `depends_on`) is what `impact` traverses by default. `related` is informational unless `--include-related` is passed.

## ID grammar

```
id := <kind> | <kind>:<scope> | <kind>:<scope>:<sub>
```

The `kind` prefix MUST match a name declared in `docs/kinds.md`. `scope` and `sub` shape is by convention per kind (see each kind's `id_pattern` field). The validator enforces only uniqueness, not pattern shape.

Concrete examples by kind:

| Kind | ID example | Notes |
|---|---|---|
| `roadmap` | `roadmap` | Singleton; no scope. |
| `brief` | `brief:roki-mvp` | One per spec. |
| `requirements` | `requirements:roki-mvp` | One per spec; carries `provides:` for the numbered requirements. |
| `req` | `req:roki-mvp:1`, `req:roki-mvp:1.6` | Declared via `provides:` of the parent requirements doc; never has its own file. |
| `design` | `design:roki-mvp`, `design:roki-mvp:bootstrap` | Spec-level main + optional sub-designs. |
| `tasks` | `tasks:roki-mvp` | One per spec. |
| `research` | `research:roki-mvp` | Optional, one per spec. |
| `fr` | `fr:01-daemon-lifecycle` | Numbered feature narrative under `docs/fr/`. |
| `reference` | `ref:cli`, `ref:config`, `ref:log-events` | Generated or hand-written reference doc under `docs/reference/`. |
| `crate` | `crate:roki-daemon` | One per crate's README. |
| `index` | `index:fr`, `index:map` | Generated by `roki-doctools index`; never hand-edited. |

## When to put which field

**`implements:`** — list every upstream artifact this doc exists to satisfy. For an `fr` doc: the `req:<spec>:N` IDs whose acceptance criteria the feature behavior covers. For a `crate` README: the spec-level `requirements:<spec>` (or relevant `req:<spec>:N`) the crate fulfills.

**`depends_on:`** — list every artifact this doc would be wrong without. For a `tasks.md`: the parent `design:<spec>`. For a sub-design (e.g. `design:roki-mvp:bootstrap`): the spec-level `design:roki-mvp`. Use sparingly — most dependencies are better expressed as `implements`.

**`related:`** — soft see-also. Use for "you should probably read this too" pointers that don't gate correctness. fr docs often `related:` other fr docs in the same spec; reference docs `related:` adjacent reference docs.

**`provides:`** — the canonical place to declare child IDs. The most common case: a `requirements.md` whose `provides:` enumerates every `req:<spec>:N` the body uses. Without an entry here, citations elsewhere become dangling references.

**`modules:`** — claim source paths that this doc documents. Use exact paths for single files and a trailing slash for directory prefixes. `touched <file>` reverses this. Listing a path here is a commitment that future changes to that file should consider whether this doc needs updating.

## Authoring rules of thumb

- **Pick the primary `id` from the kind's `id_pattern` in `docs/kinds.md`.** Don't invent new shapes.
- **Mirror nearby siblings.** When adding a new fr or new spec doc, copy the closest-shaped existing file's frontmatter and adjust.
- **`provides:` is the producer's responsibility.** The validator only catches dangling citations on the consumer side. Adding a `req:<spec>:N` citation in another doc without first listing it under the parent `requirements.md`'s `provides:` will produce a dangling-reference error.
- **`modules:` paths must exist on disk.** Renaming a source file breaks any doc whose `modules:` cites the old path. Update the doc.
- **Generated files have `generated: true` and are written by `index`.** Don't hand-edit them; re-run `index` instead.

## Common shapes

### Numbered fr doc
```yaml
---
refs:
  id: fr:NN-slug
  kind: fr
  title: "Human-readable title"
  spec: <spec-name>
  implements:
    - req:<spec>:N
    - req:<spec>:N+1
  depends_on:
    - fr:other-fr
  related:
    - ref:cli
    - design:<spec>:sub
  modules:
    - crates/<crate>/src/<area>/
---
```

### Spec requirements doc
```yaml
---
refs:
  id: requirements:<spec>
  kind: requirements
  title: "<spec> Requirements"
  spec: <spec>
  implements:
    - roadmap
  provides:
    - req:<spec>:1
    - req:<spec>:1.4    # sub-criterion if other docs cite it specifically
    - req:<spec>:2
    # ... one entry per requirement number used in the body
---
```

### Sub-design doc
```yaml
---
refs:
  id: design:<spec>:<slug>
  kind: design
  title: "<descriptive title>"
  spec: <spec>
  implements:
    - req:<spec>:N
  depends_on:
    - design:<spec>
  modules:
    - crates/<crate>/src/<file>.rs
---
```

### Crate README
```yaml
---
refs:
  id: crate:<crate-name>
  kind: crate
  title: "<crate-name>"
  spec: <spec>
  implements:
    - requirements:<spec>
  related:
    - design:<spec>
  modules:
    - crates/<crate-name>/
---
```

## What the validator does NOT enforce

- That prose `## Traceability` sections agree with the frontmatter.
- That `provides:` enumerates every requirement actually written in the body.
- ID-pattern shape (only uniqueness is enforced).
- Glob expansion in `modules:` — only literal paths and directory prefixes.

These are authoring responsibilities. Code review catches drift.
