# Requirements Document

## Project Description (Input)
Developers manually shepherd Linear tickets through implementation: read ticket, set up repo, prompt the agent, watch it work, transition Linear states, open PR. The supervision burden scales linearly with ticket volume, and humans drift between tickets while waiting on the agent. roki-mvp delivers a near-one-way Linear -> PR path with guardrails that doesn't require constant minding. It is the symphony-parity vertical slice: a `roki` Rust binary running as a daemon that polls Linear (or accepts webhooks), admits only active tickets assigned to the configured operator, launches an isolated session per admitted ticket, runs a single bounded `claude --print --output-format stream-json` invocation with kiro + superpowers skills available — the kiro skill performs all agent orchestration internally and runs the ticket end-to-end within that one invocation — and observes the agent through Linear state transitions to PR open. The daemon never writes Linear, never creates PRs, never edits code; the agent does all of that via the operator's installed Linear MCP and `gh` / `git` CLIs (reachable through Bash) inside its sandbox. The daemon configures an allowlist of repos and runs a short pre-flight setup judge — a one-shot `claude` invocation that classifies which configured repo(s) a ticket actually applies to — then materializes the corresponding git worktrees via `wt` + `ghq` itself before launching the main agent session. Cross-repo tickets fall out for free because the judge can return multiple repos. The agent inside the worker session inherits the operator's local Claude Code tool surface (Bash plus the operator's installed MCP servers, including Linear MCP for Linear reads/writes); the daemon never registers, proxies, or wraps agent-side tools. Per-issue state is keyed by Linear issue id alone. `WORKFLOW.md` (Liquid + Markdown, hot reload) is a single workspace-level policy artifact.

## Introduction

The roki-mvp specification defines the foundational vertical slice of the roki system: a single Rust daemon that watches Linear for active tickets, allocates per-issue sessions, and supervises bounded Claude Code subprocess invocations that perform the actual implementation work. The daemon is a passive observer of Linear state and an active controller of subprocess lifecycle, session filesystem state, and on-demand git worktree provisioning — it never mutates Linear, never opens pull requests, and never edits code. All write effects on Linear, GitHub, and the working tree are delegated to the agent running inside the subprocess sandbox; the daemon's view into the agent loop is a thin observation layer over stream-json output, used for structured logging, stall detection, and final-outcome classification rather than for driving the agent.

This MVP is symphony-aligned: in-memory orchestrator with no persistent database, recovery driven by re-reading Linear and the filesystem on restart, a single workspace-level `WORKFLOW.md` as the user-facing policy boundary, and a single bounded stdio agent invocation per admitted issue (assigned to the configured operator and in a Linear workflow state in the configured `admit_states` set) (the agent-side kiro skill performs all internal agent orchestration and runs the ticket end-to-end within that one invocation; the daemon does not relaunch the worker on clean exit). It diverges from symphony in being multi-repo-capable from day one through a daemon-driven setup judge (a short pre-flight Claude invocation that classifies which configured repos an admitted issue requires; the daemon then materializes worktrees and reclaims them on cleanup, both via `wt` + `ghq` against the operator's allowlist), in applying a daemon-side assignee admission filter before worker launch, in inheriting the operator's local Claude Code tool surface inside each worker (no daemon-registered or daemon-proxied agent tools), and in publishing stable extension points so dependent specs (roki-spec-gate, roki-review-gate, roki-observability, roki-distill-postmerge) can plug in without forking the orchestrator.

## Boundary Context

- **In scope**: Rust binary with CLI, async runtime, structured logging; single workspace-level `WORKFLOW.md` loader with Liquid templating, Markdown front matter, schema validation, and hot reload, exposing two named template blocks (`prompt_template_setup` for the pre-flight setup judge, `prompt_template_worker` for the main worker); in-memory state machine for per-issue worker lifecycle keyed by Linear issue id; Linear GraphQL client (read-only on the daemon side) with a single workspace-level webhook receiver and a single workspace-level polling fallback; daemon-side Linear assignee admission filter configured from the daemon configuration file (`roki.toml` by default), including the special value `me` resolved against the configured Linear API token; tracker normalization (issue model, state extraction, label extraction, assignee extraction); pre-flight setup judge — a short-lived `claude` invocation classifying admitted issues into `act` (with one or more configured repos) or `noop` (no worker required), with operator-configurable judge model (default: a small fast model from the same Claude family as the worker); daemon-driven worktree setup via `ghq list -p` + `wt switch-create` based on validated judge findings; per-issue session tempdir lifecycle (created only when the judge returns `act`, ephemeral per issue); bounded main Claude Code subprocess adapter (one invocation per admitted issue: launch, stream-json event parsing for structured logging and observability, stall detection by event-inactivity, `--max-turns` flag passthrough enforced by the subprocess, configurable ticket-level retry budget on non-clean exits); operator-declared allowlist of repos that bounds which repositories the daemon will materialize worktrees in (the allowlist is the daemon's enforcement boundary for setup-judge findings); daemon-driven worktree cleanup via allowlist iteration plus `wt remove` for any branch whose name equals the Linear issue identifier verbatim (the same discovery primitive Requirement 10 uses for restart recovery); bounded loops (`--max-turns` flag passthrough enforced by the worker subprocess, exponential backoff between ticket-level retry attempts bounded between ten seconds and five minutes, configurable retry budget on non-clean exit); configurable permission strategy (`--settings` allowlist with `--dangerously-skip-permissions` fallback); default agent sandbox set to `workspace-write` with elicitations rejected, overridable through the workspace-level `WORKFLOW.md`.
- **Out of scope**: any logic that writes Linear state, opens or comments on pull requests, or edits source files — the agent owns those effects, executed through the operator's local Claude Code tool surface; daemon-registered, daemon-proxied, or daemon-wrapped agent-side tools (the worker subprocess inherits the operator's Claude Code installation as-is, which means the agent reaches Linear via the operator's installed Linear MCP and reaches `git`/`gh`/`ghq`/`wt` via Bash); persistent state stores (SQLite or otherwise); kiro-spec gate enforcement (deferred to roki-spec-gate); kiro-review gate enforcement (deferred to roki-review-gate); HTTP API and TUI observability surfaces (deferred to roki-observability); post-merge flow-document distill sweep (deferred to roki-distill-postmerge); container or VM isolation; multi-host SSH workers; auto-merge orchestration; per-repo `WORKFLOW.md` overrides; agent-side ownership filtering as the primary admission guard; generic team, label, or project admission filtering beyond the configured assignee constraint; Windows support.
- **Adjacent expectations**: the operator installs Claude Code locally with kiro skills available as personal skills under `~/.claude/skills/kiro-*` (not vendored, not plugin-namespaced); the operator installs the `wt` (worktrunk) and `ghq` external CLIs and ensures both are on `$PATH`; the operator installs and configures a Linear MCP integration (or equivalent agent-side Linear tooling) inside their Claude Code installation, so that the worker can read and write Linear without the daemon proxying any Linear write path; the operator provides a Linear API token (used by the daemon for read-only tracker access only) and a single workspace-level Linear webhook secret; the operator declares the Linear assignee filter in the daemon configuration file (`roki.toml` by default), using `me` when roki should resolve the operator from the Linear API token; the operator declares a setup judge model identifier in configuration (or accepts the documented default); the operator configures one or more repository allowlist entries by their `ghq` identifier; the operator authors a single workspace-level `WORKFLOW.md` (with the two named template blocks) or relies on the bundled default; downstream specs (roki-spec-gate, roki-review-gate, roki-observability, roki-distill-postmerge) depend on stable state-machine hooks, the setup judge findings shape, and the workspace-level `WORKFLOW.md` schema published by this MVP.

## Requirements

### Requirement 1: Daemon Lifecycle and CLI

**Objective:** As an operator, I want a single `roki` binary that runs as a long-running daemon and exposes a clear CLI, so that I can start, configure, and stop the system without bespoke scripting.

#### Acceptance Criteria
1. When the operator invokes `roki run` with a valid configuration, the roki daemon shall start the orchestrator, the Linear adapter, the workflow loader, and the webhook server before reporting ready.
2. If the configuration file is missing, malformed, or fails schema validation, the roki daemon shall exit with a non-zero status and emit a structured log entry that names the offending field.
3. If the `wt` external CLI, the `ghq` external CLI, or the configured `claude` binary is not present at startup, the roki daemon shall refuse to start and shall log an actionable remediation message that names the missing executable.
4. When the operator sends SIGINT or SIGTERM to a running daemon, the roki daemon shall stop accepting new work, signal each active worker subprocess to terminate, await a bounded shutdown window per worker, and exit cleanly.
5. The roki daemon shall emit structured logs through a tracing pipeline that records per-issue and per-worker context fields for every event it produces, plus a per-repo field for events scoped to a specific repo (for example setup-judge worktree creation outcomes and worktree cleanup).
6. The roki daemon shall accept the CLI flags `--config <path>`, `--bind <addr>`, `--port <num>`, `--dangerously-skip-permissions`, and `--debug`, where flags override configuration values; the daemon shall document each flag in `--help` output for `roki run` and any subcommand.
7. When the operator invokes `roki --help` or any subcommand with `--help`, the roki daemon shall print usage information that documents all configuration knobs surfaced through the CLI.

### Requirement 2: Configuration, Assignee Admission, and Multi-Repo Allowlist

**Objective:** As an operator, I want to configure one daemon to serve only my assigned Linear tickets across an allowlist of Git repositories with shared Linear credentials, so that I can run a single roki instance across my whole project portfolio without consuming tickets assigned to other people.

#### Acceptance Criteria
1. The roki daemon shall accept a configuration source that declares zero or more repository allowlist entries, where each entry is identified by its `ghq` repository identifier (`owner/repo` or `host/owner/repo`) and where the local clone path is resolved at runtime through `ghq`.
2. If two configured repository allowlist entries declare the same `ghq` identifier, the roki daemon shall refuse to start and shall log the offending duplicate entry.
3. The roki daemon shall accept a single workspace-level `[linear]` configuration block declaring the Linear API token source and the Linear webhook secret source, and shall refuse to start if the Linear API token or the webhook secret cannot be resolved.
4. The roki daemon shall accept a single workspace-level `[workflow]` configuration block declaring the path to the workspace-level `WORKFLOW.md`, and shall refuse to start if that path is missing or unreadable.
5. The roki daemon shall accept a `[server]` configuration block declaring the bind address and port for the webhook receiver, and shall apply CLI overrides for both fields when supplied.
6. The roki daemon shall key all per-issue runtime state by the Linear issue identifier alone, independently of any specific repository.
7. If the configured repository allowlist is empty, the roki daemon shall start with a warning, and the setup judge's findings shall fail allowlist validation for any `act` outcome that names a repository — admitted issues effectively flow to `Skipped` until the operator configures at least one repository.
8. The roki daemon shall accept a single workspace-level Linear assignee filter in the daemon configuration file (`roki.toml` by default) under the `[linear]` configuration block, where the value `me` means the Linear user associated with the configured Linear API token.
9. If the Linear assignee filter is missing, empty, or cannot be resolved to exactly one Linear user, the roki daemon shall refuse to start and shall log the offending configuration field.
10. The roki daemon shall accept a workspace-level configuration knob declaring the setup judge model identifier; when the operator omits the override, the daemon shall apply a documented default judge model from the same Claude family as the main worker model.
11. The roki daemon shall accept a workspace-level `[linear].admit_states` configuration knob declaring the set of Linear workflow state names eligible for admission (default: `["Todo"]`); the daemon shall refuse to start if the resolved set is empty. Only issues whose current Linear workflow state name is contained in this set shall be admitted to the orchestrator; issues in any other Linear state — including states that indicate someone is already working on the ticket (e.g., `In Progress`, `In Review`) — shall not trigger judge or worker dispatch.
12. The roki daemon shall accept an optional workspace-level `[notifications.slack]` configuration block declaring at minimum a webhook URL or bot token source plus a target channel identifier; if the block is present, the daemon shall refuse to start when the destination cannot be resolved to a usable Slack endpoint, and if the block is absent, the daemon shall continue to start and run while logging a startup warning that no operator notification channel is configured.

### Requirement 3: Linear Tracker Integration

**Objective:** As an operator, I want roki to discover and track only active Linear issues assigned to the configured operator with low overhead and respect Linear's rate limits, so that the daemon stays responsive without exhausting the API quota or handling another person's work.

#### Acceptance Criteria
1. When a Linear webhook payload arrives at the single workspace-level webhook endpoint, the roki daemon shall verify the HMAC signature against the workspace-level webhook secret, normalize the payload into the internal issue model, and apply the configured Linear assignee filter before updating the orchestrator's in-memory state.
2. If a webhook payload arrives without a valid signature header or with a signature that does not match the workspace-level webhook secret, the roki daemon shall reject the request with the documented unauthorized status code without normalizing the payload.
3. While webhook delivery is unavailable, the roki daemon shall poll Linear for issues whose Linear workflow state is in the configured `admit_states` set and whose assignee matches the configured Linear assignee, on a single workspace-level cadence that does not exceed once every five minutes.
4. If Linear returns an HTTP 429 response, the roki daemon shall apply exponential backoff before its next request and shall log the backoff window.
5. The roki daemon shall expose Linear data to the rest of the system only through a normalized issue model that includes at minimum the issue identifier, title, description, current state, label set, and assignee user identifier.
6. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from the agent through the operator-configured Linear MCP integration (or equivalent agent-side tooling provided by the operator's Claude Code installation), and the daemon shall not register, proxy, or wrap any Linear write path on the agent's behalf.
7. When a webhook payload or poll result contains an issue whose Linear workflow state is in the configured `admit_states` set and whose assignee matches the configured Linear assignee, the roki daemon shall admit the issue to the orchestrator and update the per-issue state machine.
8. If a webhook payload or poll result contains an issue that fails any of the admission conditions — assignment mismatch, Linear workflow state outside `admit_states`, or both — the roki daemon shall ignore the issue for worker admission, shall not create a session tempdir, shall not launch a judge or worker subprocess, and shall emit a structured log event naming the issue identifier and the rejection reason.
9. When a previously ignored issue subsequently satisfies all admission conditions (assignee matches and Linear workflow state enters `admit_states`), the roki daemon shall admit that issue on the next valid webhook or poll observation.
10. When an already-running issue is reassigned away from the configured Linear assignee, the roki daemon shall stop further worker launches for that issue, terminate any active worker subprocess for the issue, route the issue into `Cleaning` without retry, and log that assignment loss caused the stop.
11. The roki daemon shall maintain an in-memory deduplication index keyed by Linear issue identifier that records, for every currently-tracked issue, the issue's current daemon state, the most recently observed Linear state snapshot, and any in-flight subprocess handle.
12. When a webhook payload or poll result names an issue that already has an entry in the deduplication index in any non-terminal state (`Discovered`, `Queued`, `Judging`, `Active`, `AwaitingCleanup`, `Backoff`, or `Cleaning`), the roki daemon shall not launch an additional judge or worker subprocess for that issue; the daemon shall update the recorded Linear state snapshot in place, and shall react to the new snapshot only through the explicit transitions documented in Requirements 3.10, 4.7, and 8.1 (assignment loss, terminal Linear state).
13. When startup recovery and a runtime webhook concurrently observe the same issue, the roki daemon shall serialize their effects so that at most one judge subprocess and at most one worker subprocess are in flight per Linear issue identifier at any instant.
14. When a webhook payload or poll result names an issue whose deduplication entry is in a terminal state (`Skipped` or `TerminalFailure`) and the new observation reports a Linear workflow state in the configured `admit_states` set with the assignee still matching, the roki daemon shall re-admit the issue by clearing the terminal entry and starting a fresh admission cycle from `Discovered`.

### Requirement 4: Per-Issue Session, Setup Judge, and Worktree Lifecycle

**Objective:** As an operator, I want each admitted Linear issue to pass through a daemon-controlled setup judge that decides which configured repos the issue requires, and I want the daemon — not the agent — to materialize those worktrees on entry and reclaim them on cleanup, so that the agent walks into a prepared workspace and the operator's allowlist remains the only enforcement boundary for git side-effects.

#### Acceptance Criteria
1. When the orchestrator admits an issue (transitioning it into `Judging`), the roki daemon shall invoke the workspace-level setup judge — a one-shot `claude` invocation that renders the `prompt_template_setup` block of the workspace `WORKFLOW.md` against the admitted Linear issue — and shall parse the judge's stdout as a structured findings document with at minimum an `action` discriminator (`act` or `noop`) and, when `action` is `act`, a list of repository identifiers.
2. The roki daemon shall validate every repository identifier returned by the setup judge against the configured repository allowlist; if any returned identifier is not in the allowlist, the roki daemon shall route the issue to `Skipped` with a structured log naming the offending identifier and the configured allowlist contents.
3. When the setup judge returns `action=act` with validated repository identifiers, the roki daemon shall resolve each repository's local checkout via `ghq list -p`, create a git worktree via `wt` whose branch name equals the Linear issue identifier verbatim, create the per-issue session tempdir under the platform-appropriate user cache root using the Linear issue identifier as the directory name, and expose the resulting worktree paths to the main worker subprocess's prompt template through a named template variable.
4. When the setup judge returns `action=noop`, the roki daemon shall route the issue directly to the `Skipped` terminal end without creating a session tempdir, without creating any worktree, and without launching the main worker subprocess.
5. If the setup judge fails to produce parseable findings within a configurable timeout, the roki daemon shall retry the judge once with the same input; if the second attempt also fails, the roki daemon shall route the issue to `TerminalFailure` with the raw judge output captured in the structured log.
6. The roki daemon shall reject any session or worktree path that, after sanitization, escapes its expected root, contains path traversal segments, or collides with another active worker's session.
7. The roki daemon shall enter the `Cleaning` state for an issue only when the tracker observes (via webhook or poll) one of: (a) the Linear issue transitioned to a Linear state configured as terminal (e.g., done, canceled), or (b) the Linear issue was reassigned away from the configured assignee. If a worker subprocess is still running for that issue at the moment of observation, the daemon shall terminate it before performing cleanup. While in `Cleaning`, the daemon shall iterate every entry in the configured repository allowlist, list existing worktrees in each repository's local checkout, remove every worktree via `wt remove` whose branch name equals the Linear issue identifier verbatim, and remove the session tempdir; the daemon shall not delete any branches.
8. When an issue lands in the `TerminalFailure` state, the roki daemon shall retain every created worktree, every branch, and the session tempdir so that the operator can inspect the residue.
9. If session creation, worktree creation, or worktree removal fails, the roki daemon shall mark the corresponding issue as failed, log the filesystem error with the offending path, and refuse to start additional work for that issue until the operator intervenes.

### Requirement 5: Bounded Claude Code Subprocess Adapter

**Objective:** As an operator, I want each admitted issue to be driven by a single bounded `claude --print --output-format stream-json` invocation whose lifecycle is observable, so that the daemon can supervise agent work as a thin observation layer without driving the agent loop itself; the agent-side kiro skill performs all internal orchestration and the daemon does not relaunch the worker between turns.

#### Acceptance Criteria
1. When the orchestrator promotes an issue to an active worker slot, the roki daemon shall launch a single `claude --print --output-format stream-json` subprocess in the issue's session tempdir and stream its stdout as newline-delimited JSON events.
2. While a worker subprocess is running, the roki daemon shall parse each emitted JSON event into a typed lifecycle event and shall map each event to structured logs and observability surfaces; per-issue state-machine transitions shall be driven by Linear state observed through the tracker and by subprocess exit, not by the content of stream-json events.
3. If a worker subprocess emits no events for longer than a configurable stall window, the roki daemon shall treat the worker as stalled, terminate the subprocess, and route the issue to `TerminalFailure` without further retry.
4. The roki daemon shall pass a configurable per-invocation turn budget to each worker subprocess via Claude Code's `--max-turns` flag; the subprocess is authoritative for honoring the budget, and if the subprocess's terminal `result` event indicates the turn budget was exhausted before the agent-side kiro skill reported successful completion, the roki daemon shall route the issue to `TerminalFailure` without further retry.
5. When a worker subprocess exits cleanly and its terminal `result` event reports normal completion (subtype `success`), the roki daemon shall record the completion, shall not relaunch the worker, and shall transition the issue's state machine from `Active` to `AwaitingCleanup`; subsequent transitions out of `AwaitingCleanup` (specifically into `Cleaning`) shall be driven only by tracker observations of a terminal Linear state or assignment loss, never by subprocess exit alone.
6. When a worker subprocess exits non-cleanly with any of the following causes, the roki daemon shall apply a configurable ticket-level retry budget (default three attempts, range one through ten); while remaining attempts exist the daemon shall apply exponential backoff between attempts bounded between ten seconds and five minutes, retain the session tempdir and every worktree the setup judge created across the retry, and re-launch a fresh worker subprocess for the same issue; once the retry budget is exhausted the daemon shall route the issue to `TerminalFailure`. The retryable causes are:
   (a) the process exits with a non-zero status without emitting a terminal `result` event;
   (b) the process is terminated by a signal;
   (c) the terminal `result` event reports a `subtype` documented by the daemon's compiled mapping as retryable (at MVP build time this is at minimum `error_during_execution`).
7. When the terminal `result` event reports a `subtype` not present in the daemon's compiled mapping — including any new `subtype` Claude Code adds after the daemon's build — the roki daemon shall route the issue to `TerminalFailure` immediately without consuming the ticket-level retry budget, capture the raw `subtype` value in the structured log, and trigger an operator notification per the operator notification requirements.
8. The roki daemon shall pass agent-launch flags so that kiro skills are discoverable from `~/.claude/skills/kiro-*` (no `--bare`) and shall not depend on slash commands at runtime.

### Requirement 6: Workspace-Level WORKFLOW.md Policy Loader

**Objective:** As an operator, I want a single workspace-level `WORKFLOW.md` file that defines policy in Liquid + Markdown with schema validation and hot reload, so that I can adjust agent behavior without restarting the daemon or recompiling Rust.

#### Acceptance Criteria
1. When the roki daemon starts, the WORKFLOW.md loader shall read the configured workspace-level `WORKFLOW.md`, parse its YAML or TOML front matter, identify the two named template blocks `prompt_template_setup` (consumed by the pre-flight setup judge) and `prompt_template_worker` (consumed by the main worker), and validate the result against the published schema.
2. If the workspace-level `WORKFLOW.md` fails schema validation at startup, the roki daemon shall refuse to start and shall log the validation error with the offending key path.
3. While the daemon is running, the WORKFLOW.md loader shall watch the workspace-level `WORKFLOW.md` for filesystem changes and shall re-validate the file before applying any changes.
4. If a hot-reload attempt produces a `WORKFLOW.md` that fails validation, the roki daemon shall keep the previously valid policy in effect and log the validation failure.
5. The WORKFLOW.md loader shall expose its parsed policy to the orchestrator through a stable schema shape that downstream specs can extend with additional keys without breaking existing consumers, and the loader shall round-trip unknown keys under the reserved extension namespaces without interpreting them.
6. When the roki daemon launches the setup judge or the main worker subprocess, the WORKFLOW.md loader shall render the corresponding template block (`prompt_template_setup` for the judge, `prompt_template_worker` for the worker) against the active Linear issue with the issue's identifier, title, description, label list, and bucketed lifecycle state available as named variables, and the roki daemon shall additionally expose the validated worktree paths as a named template variable to `prompt_template_worker`; the rendered output shall become the corresponding subprocess's prompt input.
7. If a workspace-level prompt template fails to render at launch, the roki daemon shall log the render failure with the offending block name and shall provide a deterministic fallback prompt that includes the Linear issue identifier, title, and description so that the subprocess always receives non-empty issue context.

### Requirement 7: Agent Tooling Boundary

**Objective:** As an operator, I want the daemon to neither register nor proxy any agent-side tool, so that the worker subprocess inherits exactly the tool surface my local Claude Code installation exposes (Bash plus my installed MCP servers) and the daemon never needs to hold Linear or git write credentials on the agent's behalf.

#### Acceptance Criteria
1. The roki daemon shall not register, proxy, or wrap any agent-side tool inside the worker subprocess; the worker subprocess's tool surface shall be exactly what the operator's local Claude Code installation exposes — its built-in tool set plus any MCP servers the operator has installed — and the daemon shall add nothing to it.
2. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from the agent through the operator-configured Linear MCP integration or equivalent agent-side Linear tooling, and the daemon shall not hold any agent-facing Linear write path.
3. Where the workspace-level `WORKFLOW.md` prompt instructs the agent to operate `git`, `gh`, `ghq`, `wt`, or Linear tooling from inside the worker session, the operator's local Claude Code installation shall be the source of those affordances (Bash for shell tools and the operator's installed MCP servers for Linear); the roki daemon shall not substitute, intercept, or augment those tool invocations.
4. The roki daemon shall never embed the Linear API token or the Linear webhook secret in any prompt input, log line, or other artifact reachable from inside the worker subprocess.

### Requirement 8: Orchestrator State Machine and Extension Points

**Objective:** As a downstream spec author, I want the orchestrator to expose stable per-issue state-machine hooks, so that roki-spec-gate, roki-review-gate, roki-observability, and roki-distill-postmerge can subscribe to lifecycle transitions without forking the core.

#### Acceptance Criteria
1. The roki daemon shall maintain an in-memory state machine per Linear issue identifier covering at minimum the following daemon-local states: `Discovered`, `Queued`, `Judging`, `Active`, `AwaitingCleanup`, `Backoff`, `Cleaning`, `Skipped`, and `TerminalFailure`. These states represent the daemon's bookkeeping for subprocess lifecycle and cleanup; the daemon does not mirror Linear's workflow states (review, done, etc.), which it observes through the tracker rather than maintaining internally. `Skipped` shall be reachable only from `Judging`, either when the setup judge returns `action=noop` or when the judge's findings fail repository allowlist validation (Req 4.2). `AwaitingCleanup` shall be reachable only from `Active` on a subprocess clean exit reporting successful completion. `Backoff` is the retry-window state for ticket-level retries after a non-clean subprocess exit. `Cleaning` shall be reachable only from `Active`, `AwaitingCleanup`, or `Backoff`, and only as a result of a tracker observation (the issue moved to a configured terminal Linear state, or the issue was reassigned away from the configured assignee); a subprocess exit alone shall never trigger entry to `Cleaning`.
2. When a state transition occurs, the roki daemon shall publish a structured transition event whose payload identifies the previous state, the next state, the trigger source, and the issue identifier, and shall include the originating repository identifier when one applies (for example for transitions caused by setup-judge worktree creation outcomes or worktree cleanup arcs).
3. The roki daemon shall expose subscription hooks that allow other components to observe transition events and to veto a specific set of declared-as-vetoable transitions, and shall document which transitions are vetoable.
4. If a subscriber raises an unhandled error while processing a transition event, the roki daemon shall isolate the failure to that subscriber, log the error with the subscriber's identifier, and continue processing transitions for other subscribers.
5. The roki daemon shall recover the per-issue state on restart by re-reading Linear and the filesystem layout, without depending on a persistent database.

### Requirement 9: Permission Strategy and Default Sandbox

**Objective:** As an operator, I want roki to run agents under the safest workable permission profile while letting me opt into a fallback when Claude's allowlist is unreliable, so that I can balance safety against operability.

#### Acceptance Criteria
1. The roki daemon shall launch each worker subprocess with the agent sandbox set to `workspace-write` and with elicitations rejected by default.
2. When the workspace-level `WORKFLOW.md` declares an alternative sandbox or elicitation policy, the roki daemon shall apply that override to every worker.
3. When the operator selects the `--settings` allowlist permission strategy, the roki daemon shall pass the configured allowlist to each worker subprocess through Claude Code's settings interface.
4. When the operator selects the `--dangerously-skip-permissions` fallback strategy through configuration or the `--dangerously-skip-permissions` CLI flag, the roki daemon shall pass that flag to each worker subprocess and shall log the elevated-permission decision per worker launch.
5. If neither permission strategy is configured, the roki daemon shall refuse to start and shall report the missing configuration.
6. When the roki daemon invokes the setup judge, the daemon shall launch the judge subprocess with a read-only sandbox and rejected elicitations regardless of operator overrides; the operator-configured permission strategy shall apply only to the main worker subprocess.

### Requirement 10: Restart Recovery Without Persistent Storage

**Objective:** As an operator, I want roki to recover its working state after a restart by re-reading Linear and the filesystem rather than relying on a database, so that I never carry stale or corrupted local state across restarts.

#### Acceptance Criteria
1. When the roki daemon starts, it shall list every session tempdir under the platform-appropriate user cache root and, for every configured allowlisted repository, list every existing git worktree whose branch name matches the operator-configurable Linear issue-identifier pattern.
2. The roki daemon shall reconcile every distinct issue identifier discovered from session tempdirs and from worktree branches against Linear before resuming work, shall apply the configured Linear assignee filter during reconciliation, and shall classify each discovered issue into one of the documented recovery states (resume-active, orphaned-session, orphaned-worktree, fresh-queued, no-op).
3. If a session tempdir or worktree exists for an issue identifier that has no matching active Linear issue assigned to the configured Linear assignee at startup, the roki daemon shall mark that residue as orphaned and log it without deleting it automatically.
4. If a Linear issue is assigned to the configured Linear assignee and is in a Linear workflow state contained in the configured `admit_states` set at startup, but has no matching session tempdir or worktree, the roki daemon shall enqueue the issue for a fresh admission cycle starting from `Discovered`.
5. The roki daemon shall not write any per-issue runtime state to disk except the session tempdir and worktree contents the agent itself produces and the structured logs the daemon emits.

### Requirement 11: Observability of Daemon Internals

**Objective:** As an operator debugging a stuck ticket, I want enough structured observability from the daemon itself to diagnose worker, session, worktree, and tracker issues without an external UI, so that the MVP is operable before the dedicated observability spec ships.

#### Acceptance Criteria
1. The roki daemon shall emit a structured log event for every worker lifecycle change, every session tempdir creation or deletion, every worktree creation or removal, every Linear poll or webhook receipt, every backoff or stall decision, every retry attempt with its attempt counter, and every state-machine transition.
2. The roki daemon shall include the Linear issue identifier when one applies, the repository identifier when one applies (for example for worktree-scoped events), and a correlation identifier for the originating worker invocation in every structured log event.
3. The roki daemon shall support a configurable log level and a configurable log destination so that the operator can route logs to stdout, a file, or both.
4. The roki daemon shall redact the Linear API token, the Linear webhook secret, and any other operator-declared secrets from every structured log event before emission.
5. While a worker subprocess or the setup judge subprocess is running, the roki daemon shall capture every line emitted on the subprocess's standard error stream and shall surface each non-empty line as a structured log event at warn severity tagged with the issue identifier, the subprocess role (`judge` or `worker`), and the correlation identifier.
6. Where the operator opts into per-issue debug capture through the documented CLI flag or configuration block, the roki daemon shall append every line emitted on each worker subprocess's standard output and standard error streams to a per-issue file under the configured debug-log directory, and each appended line shall carry an RFC 3339 timestamp with nanosecond resolution and a stream tag identifying standard output or standard error.
7. If the per-issue debug log file cannot be opened or appended to, the roki daemon shall log the failure with the offending file path and shall continue running the worker without aborting the launch.
8. When the setup judge completes (success, retry, or final failure), the roki daemon shall emit a structured log event recording the judge duration, the parsed `action` discriminator (when parseable), the validated repositories or rejection reason, and the issue identifier.

### Requirement 12: Operator Notification Channel

**Objective:** As an operator, I want roki to surface daemon-level failures that the agent inside the worker subprocess cannot self-report, into a separate channel that I actually monitor, so that I learn about stuck or failed tickets without continuously tailing daemon logs and without polluting the Linear tickets with roki-originated noise.

#### Acceptance Criteria
1. The roki daemon shall use the workspace-level `[notifications.slack]` configuration block declared in Requirement 2.12 as the destination for every operator notification this requirement defines; when that block is absent, the daemon shall skip Slack posting at every notification site while continuing to emit the corresponding structured log event.
2. The roki daemon shall post an operator notification to the configured Slack channel when any of the following daemon-level events occurs, because the worker subprocess either could not write to Linear or because the event is generated by the daemon itself rather than by the agent:
   (a) an issue is routed to `TerminalFailure` after stall detection (Requirement 5.3) — the daemon killed the subprocess so the agent had no chance to self-report;
   (b) an issue is routed to `TerminalFailure` after `--max-turns` exhaustion (Requirement 5.4) — Claude Code terminated the agent before it could finish its current turn or write a final report;
   (c) an issue is routed to `TerminalFailure` because the terminal `result` event reported an unknown `subtype` (Requirement 5.7);
   (d) an issue is routed to `TerminalFailure` after the ticket-level retry budget is exhausted (Requirement 5.6);
   (e) the setup judge fails to produce parseable findings even after the documented retry (Requirement 4.5) — the judge has no Linear write capability;
   (f) a filesystem error during session or worktree creation, removal, or rename poisons an issue (Requirement 4.9);
   (g) startup recovery identifies an orphaned session tempdir or worktree retained for inspection (Requirement 10.3).
3. The roki daemon shall not post operator notifications to Slack for events that the worker subprocess is expected to self-report through Linear, including normal completions and any agent-recoverable error the agent surfaces in the ticket itself.
4. Each Slack notification shall include at minimum the Linear issue identifier with a Linear-resolvable link, the daemon-internal failure category, the correlation identifier of the originating worker invocation when one applies, the repository identifier when one applies, and the event timestamp; the notification shall not include the Linear API token, the Linear webhook secret, the Slack webhook URL or bot token, or any other operator-declared secret.
5. If posting to Slack fails (network error, non-success HTTP status, configuration drift), the roki daemon shall log the failure with the destination identifier and the underlying error, shall apply at most one immediate retry, and shall not crash, block, or alter the per-issue state machine on persistent failure.

### Requirement 13: Cross-Spec Extension Surface

**Objective:** As a downstream spec author, I want roki-mvp to publish a stable, additive extension surface, so that roki-spec-gate, roki-review-gate, roki-observability, and roki-distill-postmerge can integrate without forking the core or breaking sibling specs.

#### Acceptance Criteria
1. The roki daemon shall publish a read-only `OrchestratorRead` trait that exposes a snapshot of the per-issue state plus a single-issue lookup, and shall grant no state-mutation rights through that trait.
2. The roki daemon shall expose a vetoable pre-cleanup hook between terminal success and worktree-and-session cleanup so that downstream specs can perform deferred work while the worker's worktrees and session tempdir still exist; a `Deny` decision shall block cleanup and shall be logged.
3. The roki tracker adapter shall publish a `TrackerRefresh` nudge trait that allows external callers to request an out-of-cycle poll without bypassing the documented cadence cap or the 429 backoff state.
4. The roki engine adapter shall accept an additive optional `additional_context` field on `WorkerContext` and shall forward its value verbatim to the agent through a stable, machine-extractable section of the worker subprocess's prompt input, kept distinct from the rendered `prompt_template_worker` body, without the daemon interpreting the contents. The exact serialization shape (delimiter sentinels, JSON envelope, named keys) is defined by the engine adapter design and is treated as additive: new optional keys may be added under the same forwarding contract; removing or retyping existing keys is a breaking change.
5. The workspace-level `WORKFLOW.md` schema shall reserve the `extension.gates.spec.*`, `extension.gates.review.*`, `extension.server.*`, and `extension.distill.*` sub-namespaces for the canonical roki specs; the loader shall round-trip unknown keys under those namespaces without interpreting them.
