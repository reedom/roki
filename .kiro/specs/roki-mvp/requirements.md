# Requirements Document

## Project Description (Input)
Developers manually shepherd Linear tickets through implementation: read ticket, set up repo, prompt the agent, watch it work, transition Linear states, open PR. The supervision burden scales linearly with ticket volume, and humans drift between tickets while waiting on the agent. roki-mvp delivers a near-one-way Linear -> PR path with guardrails that doesn't require constant minding. It is the symphony-parity vertical slice: a `roki` Rust binary running as a daemon that polls Linear (or accepts webhooks), admits only active tickets assigned to the configured operator, and per admitted ticket launches a long-lived **orchestrator session** — a `claude --input-format stream-json --output-format stream-json` "thinking" component (A) whose tool surface is restricted to the operator's installed Linear MCP (write) plus `Read` and `Bash` (read-only filesystem sandbox) and whose responses are strict JSON action directives over a small enum. A absorbs admission classification, phase planning, artifact validation (`requirements.md` after `materialize_spec`, `review.md` after `finalize_review`), daemon-only failure surfacing, and Linear writes; the daemon never writes Linear directly and never decides whether an artifact passes. For each phase A nominates, the daemon spawns a short-lived bounded `claude -p '/kiro-* <args>' --output-format stream-json` subprocess driven by a slash-command kiro skill that performs the actual code-changing work. Phase catalog: `materialize_spec` (kiro-discovery, produces per-issue `requirements.md`), `implement` (kiro-impl), `validate` (kiro-validate-impl), `open_pr` (custom prompt, no skill), `ci_fix` (kiro-debug + kiro-verify-completion), `finalize_review` (review.md synthesis). The daemon configures an allowlist of repos. A admits or rejects an issue (`act` / `noop` / `needs_split` / `allowlist_rejected`); the daemon validates that A's classification names exactly one allowlisted repo and then materializes the corresponding git worktree via `wt` + `ghq`. The agent inside every phase subprocess inherits the operator's local Claude Code tool surface (Bash plus the operator's installed MCP servers, including Linear MCP); the daemon never registers, proxies, or wraps agent-side tools. Per-issue state is keyed by Linear issue id alone. `WORKFLOW.md` (Liquid + Markdown, hot reload) is a single workspace-level policy artifact whose only named template block is `prompt_template_orchestrator`.

## Introduction

The roki-mvp specification defines the foundational vertical slice of the roki system: a single Rust daemon that watches Linear for active tickets, allocates per-issue sessions, and supervises two shapes of bounded Claude Code subprocess invocations that perform the actual implementation work. The daemon is a passive observer of Linear state and an active controller of subprocess lifecycle, session filesystem state, and on-demand git worktree provisioning — it never mutates Linear, never opens pull requests, and never edits code. All write effects on Linear, GitHub, and the working tree are delegated to agents running inside subprocess sandboxes; the daemon's view into each agent loop is a thin observation layer over stream-json output, used for structured logging, stall detection, schema-drift detection, and final-outcome classification rather than for driving the agent.

This MVP is symphony-aligned: in-memory orchestrator with no persistent database, recovery driven by re-reading Linear and the filesystem on restart, a single workspace-level `WORKFLOW.md` as the user-facing policy boundary, and a two-shape engine taxonomy per admitted issue. The two shapes are: (a) one long-lived **orchestrator session** A (`claude --input-format stream-json --output-format stream-json`) that emits strict JSON action directives, absorbing the admission decision (replacing the prior setup-judge subprocess), the daemon-only failure surfacing channel (replacing the prior linear-updater subagent), and artifact validation (replacing the prior daemon-side mechanical kiro-spec / kiro-review gates); and (b) zero or more short-lived bounded **phase subprocesses** (`claude -p '/kiro-* <args>' --output-format stream-json`) that A nominates via `run_phase`, each driving a slash-command-based kiro skill in headless mode. roki diverges from symphony in being daemon-driven on workspace lifecycle (the daemon — not A — materializes and reclaims the worktree, both via `wt` + `ghq` against the operator's allowlist), in applying a daemon-side assignee admission filter before A is launched, in inheriting the operator's local Claude Code tool surface inside each phase subprocess (no daemon-registered or daemon-proxied agent tools), and in publishing stable extension points so dependent specs (roki-observability) can plug in without forking the orchestrator.

A's full lifecycle, event catalog, response schema, tool surface, configuration, artifact-validation contract, and failure modes are canonicalized in [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). This requirements document references that contract rather than restating it.

## Boundary Context

- **In scope**: Rust binary with CLI, async runtime, structured logging; single workspace-level `WORKFLOW.md` loader with Liquid templating, Markdown front matter, schema validation, and hot reload, exposing one named template block (`prompt_template_orchestrator`); in-memory state machine for per-issue lifecycle keyed by Linear issue id (six states with an `Inactive` reason discriminator that includes `orchestrator_crash`, `orchestrator_unparseable`, `orchestrator_budget_exhausted`); Linear GraphQL client (read-only on the daemon side) with a single workspace-level webhook receiver and a single workspace-level polling fallback; daemon-side Linear assignee admission filter configured from the daemon configuration file (`roki.toml` by default), including the special value `me` resolved against the configured Linear API token; tracker normalization (issue model, state extraction, label extraction, assignee extraction); long-lived orchestrator session A per admitted issue (one `claude --input-format stream-json --output-format stream-json` invocation, launched on `Discovered → Pending`, configured under the reserved `extension.orchestrator.{model, effort, max_phases, allowed_tools}` namespace in `WORKFLOW.md`, runs with a read-only filesystem sandbox regardless of operator overrides); A-driven admission classification via the `admission_request` event (A returns `judge: act | noop | needs_split | allowlist_rejected`) with daemon-side validation that an `act` classification names exactly one allowlisted repo; daemon-driven worktree setup via `ghq list -p` + `wt switch-create` once A returns `judge: act` for a single allowlisted repo; per-issue session tempdir lifecycle (created only when A returns `judge: act` for a validated single allowlisted repo, ephemeral per issue); A-driven artifact validation (after each `materialize_spec` clean exit A reads `requirements.md` and decides next phase or retry; after each `finalize_review` clean exit A reads `review.md` and decides `action=stop` or retry, using `Read` + `Bash` only inside its read-only filesystem sandbox); short-lived bounded phase subprocesses for code-changing work — one `claude -p '/kiro-* <args>' --output-format stream-json` per phase that A nominates, each with its own `--max-turns` budget, slash-command-driven kiro skill, stream-json parsing, and stall detection; phase catalog `materialize_spec` (kiro-discovery, produces per-issue `requirements.md`), `implement` (kiro-impl), `validate` (kiro-validate-impl), `open_pr` (custom prompt, no skill), `ci_fix` (kiro-debug + kiro-verify-completion), `finalize_review` (review.md synthesis); A processes `daemon_directive` events for daemon-only failure surfacing (stall, retry exhaustion, fs poison, orphan recovery, multi-repo rejection, etc.) and writes labels + comments to Linear via the operator's installed Linear MCP within its own session — the daemon never issues a Linear write itself; operator-declared allowlist of repos that bounds which repositories the daemon will materialize worktrees in (the allowlist is the daemon's enforcement boundary for A's classification, and any classification naming more than one repo is rejected back through A via the `needs_split` rejection path); daemon-driven worktree cleanup via allowlist iteration plus `wt remove` for any branch whose name equals the Linear issue identifier verbatim (the same discovery primitive Requirement 10 uses for restart recovery); bounded loops (per-phase `--max-turns` flag passthrough enforced by the phase subprocess; `max_phases` budget enforced by the daemon for A's phase nominations; exponential backoff between ticket-level retry attempts bounded between ten seconds and five minutes on phase non-clean exit; configurable retry budget on phase non-clean exit); configurable permission strategy (`--settings` allowlist with `--dangerously-skip-permissions` fallback) for phase subprocesses; default phase-subprocess sandbox set to `workspace-write` with elicitations rejected, overridable through the workspace-level `WORKFLOW.md`; A always runs with a read-only filesystem sandbox regardless of operator overrides, with `Bash` permitted but unable to mutate the filesystem due to that sandbox.
- **Out of scope**: any logic that writes Linear state, opens or comments on pull requests, or edits source files — Linear writes originate exclusively from A through the operator's installed Linear MCP, and code / PR effects originate exclusively from phase subprocesses through the operator's local Claude Code tool surface; daemon-registered, daemon-proxied, or daemon-wrapped agent-side tools (every subprocess inherits the operator's Claude Code installation as-is, narrowed only by per-process `allowed_tools`); persistent state stores (SQLite or otherwise); daemon-side mechanical artifact-validation gates (the prior roki-spec-gate / roki-review-gate vetoable hooks are removed; A reads `requirements.md` and `review.md` itself per its own retry-budget rules); HTTP API and TUI observability surfaces (deferred to roki-observability); post-merge flow-document distill sweep (handled in CI, not by the daemon); container or VM isolation; multi-host SSH workers; auto-merge orchestration; per-repo `WORKFLOW.md` overrides; multi-repo tickets (one ticket = one repo; multi-repo classifications are rejected by A via the `needs_split` admission decision); agent-side ownership filtering as the primary admission guard; generic team, label, or project admission filtering beyond the configured assignee constraint; Slack and other push-style operator notification channels (daemon-only failures surface via A → Linear MCP when A is alive, and via TUI escalation queue + structured log only for the three orchestrator-dead failure paths); A-side context compaction across long-running tickets (deferred — the `max_phases` budget bounds session length for MVP); Rust-native read-only artifact-validation tool to replace A's `Bash` use (deferred — for MVP A uses Bash directly); Windows support.
- **Adjacent expectations**: the operator installs Claude Code locally with kiro skills available as personal skills under `~/.claude/skills/kiro-*` (not vendored, not plugin-namespaced); the operator installs the `wt` (worktrunk) and `ghq` external CLIs and ensures both are on `$PATH`; the operator installs and configures a Linear MCP integration (or equivalent agent-side Linear tooling) inside their Claude Code installation, so that A and phase subprocesses can read and write Linear without the daemon proxying any Linear write path; the operator provides a Linear API token (used by the daemon for read-only tracker access only) and a single workspace-level Linear webhook secret; the operator declares the Linear assignee filter in the daemon configuration file (`roki.toml` by default), using `me` when roki should resolve the operator from the Linear API token; the operator configures one or more repository allowlist entries by their `ghq` identifier; the operator authors a single workspace-level `WORKFLOW.md` (with the named template block `prompt_template_orchestrator`) or relies on the bundled default; downstream spec (roki-observability) depends on stable state-machine hooks, A's response schema, and the workspace-level `WORKFLOW.md` schema published by this MVP.

## Requirements

### Requirement 1: Daemon Lifecycle and CLI

**Objective:** As an operator, I want a single `roki` binary that runs as a long-running daemon and exposes a clear CLI, so that I can start, configure, and stop the system without bespoke scripting.

#### Acceptance Criteria
1. When the operator invokes `roki run` with a valid configuration, the roki daemon shall start the orchestrator, the Linear adapter, the workflow loader, and the webhook server before reporting ready.
2. If the configuration file is missing, malformed, or fails schema validation, the roki daemon shall exit with a non-zero status and emit a structured log entry that names the offending field.
3. If the `wt` external CLI, the `ghq` external CLI, or the configured `claude` binary is not present at startup, the roki daemon shall refuse to start and shall log an actionable remediation message that names the missing executable.
4. When the operator sends SIGINT or SIGTERM to a running daemon, the roki daemon shall stop accepting new work, signal each active orchestrator session and each active phase subprocess to terminate, await a bounded shutdown window per subprocess, and exit cleanly.
5. The roki daemon shall emit structured logs through a tracing pipeline that records per-issue and per-subprocess context fields for every event it produces, plus a per-repo field for events scoped to a specific repo (for example worktree creation outcomes and worktree cleanup). The structured event catalog is canonical in [`docs/reference/log-events.md`](../../../docs/reference/log-events.md).
6. The roki daemon shall accept the CLI flags `--config <path>`, `--bind <addr>`, `--port <num>`, `--dangerously-skip-permissions`, and `--debug`, where flags override configuration values; the daemon shall document each flag in `--help` output for `roki run` and any subcommand. The canonical CLI surface is [`docs/reference/cli.md`](../../../docs/reference/cli.md).
7. When the operator invokes `roki --help` or any subcommand with `--help`, the roki daemon shall print usage information that documents all configuration knobs surfaced through the CLI.

### Requirement 2: Configuration and Assignee Admission

**Objective:** As an operator, I want to configure one daemon to serve only my assigned Linear tickets across an allowlist of Git repositories with shared Linear credentials, so that I can run a single roki instance across my whole project portfolio without consuming tickets assigned to other people.

#### Acceptance Criteria
1. The roki daemon shall accept a configuration source that declares zero or more repository allowlist entries, where each entry is identified by its `ghq` repository identifier (`owner/repo` or `host/owner/repo`) and where the local clone path is resolved at runtime through `ghq`.
2. If two configured repository allowlist entries declare the same `ghq` identifier, the roki daemon shall refuse to start and shall log the offending duplicate entry.
3. The roki daemon shall accept a single workspace-level `[linear]` configuration block declaring the Linear API token source and the Linear webhook secret source, and shall refuse to start if the Linear API token or the webhook secret cannot be resolved.
4. The roki daemon shall accept a single workspace-level `[workflow]` configuration block declaring the path to the workspace-level `WORKFLOW.md`, and shall refuse to start if that path is missing or unreadable.
5. The roki daemon shall accept a `[server]` configuration block declaring the bind address and port for the webhook receiver, and shall apply CLI overrides for both fields when supplied.
6. The roki daemon shall key all per-issue runtime state by the Linear issue identifier alone, independently of any specific repository.
7. If the configured repository allowlist is empty, the roki daemon shall start with a warning, and any subsequent A-returned `act` admission decision shall fail allowlist validation — affected admitted issues effectively flow to `Inactive(reason=allowlist_rejected)` until the operator configures at least one repository.
8. The roki daemon shall accept a single workspace-level Linear assignee filter in the daemon configuration file (`roki.toml` by default) under the `[linear]` configuration block, where the value `me` means the Linear user associated with the configured Linear API token.
9. If the Linear assignee filter is missing, empty, or cannot be resolved to exactly one Linear user, the roki daemon shall refuse to start and shall log the offending configuration field.
10. The roki daemon shall accept a workspace-level `[linear].admit_states` configuration knob declaring the set of Linear workflow state names eligible for admission (default: `["Todo"]`); the daemon shall refuse to start if the resolved set is empty. Only issues whose current Linear workflow state name is contained in this set shall be admitted to the orchestrator; issues in any other Linear state — including states that indicate someone is already working on the ticket (e.g., `In Progress`, `In Review`) — shall not trigger A launch.
11. The roki daemon shall reserve the `extension.orchestrator.*` configuration namespace under the workspace-level `WORKFLOW.md` for orchestrator session A; the namespace shall accept the keys `model`, `effort`, `max_phases`, and `allowed_tools` per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). When any of these keys is omitted, the daemon shall apply the canonical defaults documented in that FR (`model = "claude-opus-4-7"`, `effort = "middle"`, `max_phases = 20`, `allowed_tools` permitting Linear MCP write + `Read` + `Bash`). If any reserved key under that namespace fails type, enum, or range validation at startup or hot reload, the daemon shall refuse the new policy at startup or retain the previous policy at hot reload, and shall log the offending key path.
12. The roki daemon shall not accept a `[judge].model` configuration key, an `extension.linear_updater.*` namespace, or any other configuration surface tied to the removed setup-judge or linear-updater subprocess shapes; the loader shall log an actionable error if any such key is observed in the configuration source so that operators upgrading from a prior architecture can migrate.
13. The canonical configuration key reference is [`docs/reference/config.md`](../../../docs/reference/config.md); this requirement document does not restate per-key validation rules already documented there.

### Requirement 3: Linear Tracker Integration

**Objective:** As an operator, I want roki to discover and track only active Linear issues assigned to the configured operator with low overhead and respect Linear's rate limits, so that the daemon stays responsive without exhausting the API quota or handling another person's work.

#### Acceptance Criteria
1. When a Linear webhook payload arrives at the single workspace-level webhook endpoint, the roki daemon shall verify the HMAC signature against the workspace-level webhook secret, normalize the payload into the internal issue model, and apply the configured Linear assignee filter before updating the orchestrator's in-memory state.
2. If a webhook payload arrives without a valid signature header or with a signature that does not match the workspace-level webhook secret, the roki daemon shall reject the request with the documented unauthorized status code without normalizing the payload.
3. While webhook delivery is unavailable, the roki daemon shall poll Linear for issues whose Linear workflow state is in the configured `admit_states` set and whose assignee matches the configured Linear assignee, on a single workspace-level cadence that does not exceed once every five minutes.
4. If Linear returns an HTTP 429 response, the roki daemon shall apply exponential backoff before its next request and shall log the backoff window.
5. The roki daemon shall expose Linear data to the rest of the system only through a normalized issue model that includes at minimum the issue identifier, title, description, current state, label set, and assignee user identifier.
6. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from orchestrator session A (through the operator-configured Linear MCP integration) or from a phase subprocess (through the same MCP, where its `allowed_tools` permits it), and the daemon shall not register, proxy, or wrap any Linear write path on the agent's behalf.
7. When a webhook payload or poll result contains an issue whose Linear workflow state is in the configured `admit_states` set and whose assignee matches the configured Linear assignee, the roki daemon shall admit the issue to the orchestrator and update the per-issue state machine.
8. If a webhook payload or poll result contains an issue that fails any of the admission conditions — assignment mismatch, Linear workflow state outside `admit_states`, or both — the roki daemon shall ignore the issue, shall not create a session tempdir, shall not launch orchestrator session A or any phase subprocess, and shall emit a structured log event naming the issue identifier and the rejection reason.
9. When a previously ignored issue subsequently satisfies all admission conditions (assignee matches and Linear workflow state enters `admit_states`), the roki daemon shall admit that issue on the next valid webhook or poll observation.
10. When an already-running issue is reassigned away from the configured Linear assignee, the roki daemon shall stop further phase nominations for that issue, terminate any active phase subprocess and orchestrator session A for the issue, route the issue into `Cleaning` without retry, and log that assignment loss caused the stop.
11. The roki daemon shall maintain an in-memory deduplication index keyed by Linear issue identifier that records, for every currently-tracked issue, the issue's current daemon state, the most recently observed Linear state snapshot, and any in-flight subprocess handle (orchestrator session and phase subprocess).
12. When a webhook payload or poll result names an issue that already has an entry in the deduplication index in any non-terminal state (`Pending`, `Judging`, `Active`, `Backoff`, or `Cleaning`), the roki daemon shall not launch an additional orchestrator session A or phase subprocess for that issue; the daemon shall update the recorded Linear state snapshot in place, and shall react to the new snapshot only through the explicit transitions documented in Requirements 3.10, 4.6, and 8.1 (assignment loss, terminal Linear state).
13. When startup recovery and a runtime webhook concurrently observe the same issue, the roki daemon shall serialize their effects so that at most one orchestrator session A and at most one phase subprocess are in flight per Linear issue identifier at any instant.
14. When a webhook payload or poll result names an issue whose deduplication entry is in the `Inactive` state and the new observation reports a Linear workflow state in the configured `admit_states` set with the assignee still matching, the roki daemon shall re-admit the issue by clearing the `Inactive` entry and starting a fresh admission cycle from `Pending`.

### Requirement 4: Per-Issue Session and Worktree Lifecycle

**Objective:** As an operator, I want each admitted Linear issue to pass through orchestrator session A's admission classification with daemon-side allowlist validation, and I want the daemon — not the agent — to materialize the worktree on entry and reclaim it on cleanup, so that A walks into a prepared workspace and the operator's allowlist remains the only enforcement boundary for git side-effects. The admission decision itself is A's responsibility per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md), not a separate setup-judge subprocess.

#### Acceptance Criteria
1. When the orchestrator admits an issue (transitioning it to `Pending`), the roki daemon shall launch orchestrator session A for the issue per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md). The daemon shall not launch any separate setup-judge subprocess — the admission decision is A's responsibility.
2. On the `Pending → Judging` transition the roki daemon shall send an `admission_request` event to A's stdin and shall parse A's response per the schema in [fr:19-orchestrator-session > Response schema](../../../docs/fr/19-orchestrator-session.md). The expected `action` is `admission_decision` with `judge ∈ {act, noop, needs_split, allowlist_rejected}`.
3. The roki daemon shall validate A's `admission_decision` as follows: (a) when `judge = act`, the response must name exactly one repository identifier in the `repo` field and that identifier must be in the configured allowlist — if the response names zero or more than one repository, or names a repository absent from the allowlist, the daemon shall override the decision and route the issue to `Inactive(reason=allowlist_rejected)` with a structured log event capturing A's raw response; (b) when `judge ∈ {needs_split, allowlist_rejected}`, A is responsible for writing the matching Linear label + comment via Linear MCP in the same turn (per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md)), after which the daemon routes the issue to `Inactive(reason=needs_split)` or `Inactive(reason=allowlist_rejected)` respectively; (c) when `judge = noop`, the daemon routes the issue directly to `Inactive(reason=noop)` without creating a session tempdir or worktree.
4. When A returns `judge = act` with a single validated allowlisted repository identifier, the roki daemon shall resolve the repository's local checkout via `ghq list -p`, create a single git worktree via `wt` whose branch name equals the Linear issue identifier verbatim, create the per-issue session tempdir under the platform-appropriate user cache root using the Linear issue identifier as the directory name, and expose the resulting worktree path to subsequent phase subprocess invocations through their per-phase context envelope.
5. If A's response to `admission_request` fails JSON schema validation, the roki daemon shall reprompt A once per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md); on a second consecutive schema-drift the daemon shall route the issue to `Inactive(reason=orchestrator_unparseable)` (no separate `judge_unparseable` reason — A is the judge).
6. The roki daemon shall reject any session or worktree path that, after sanitization, escapes its expected root, contains path traversal segments, or collides with another active issue's session.
7. The roki daemon shall enter the `Cleaning` state for an issue only when the tracker observes (via webhook or poll) one of: (a) the Linear issue transitioned to a Linear state configured as terminal (e.g., done, canceled), or (b) the Linear issue was reassigned away from the configured assignee. If orchestrator session A or any phase subprocess is still running for that issue at the moment of observation, the daemon shall terminate them before performing cleanup. While in `Cleaning`, the daemon shall locate the worktree whose branch name equals the Linear issue identifier verbatim under the configured allowlisted repository's local checkout, remove it via `wt remove`, and remove the session tempdir; the daemon shall not delete any branches.
8. When an issue lands in any `Inactive(reason=*)` failure state (any non-`noop`, non-`awaiting_linear` reason), the roki daemon shall retain the worktree, the branch, and the session tempdir so that the operator can inspect the residue, and shall not auto-clean until the tracker observes the issue's Linear state moving to a configured terminal state.
9. If session creation, worktree creation, or worktree removal fails, the roki daemon shall mark the corresponding issue as `Inactive(reason=fs_poison)`, log the filesystem error with the offending path, send a `daemon_directive` event with kind `fs_poison` to A (when A is still alive) so that A surfaces the failure to Linear, and refuse to start additional work for that issue until the operator intervenes. If A is no longer alive at the moment of `fs_poison` (it has already exited or been force-terminated), the failure surfaces via structured log + TUI escalation queue only, per Requirement 12.

### Requirement 5: Bounded Subprocess Adapters (Orchestrator Session A and Phase Subprocesses)

**Objective:** As an operator, I want each admitted issue to be driven by a long-lived orchestrator session A whose JSON action directives nominate short-lived bounded phase subprocesses for the actual code-changing work, so that the daemon supervises both shapes as a thin observation layer with bounded thinking effort (`max_phases`) on A and bounded turn budgets (`--max-turns`) on each phase, without driving any agent loop itself.

The two engine shapes the daemon supervises per ticket are: (a) **orchestrator session A** — one long-lived `claude --input-format stream-json --output-format stream-json` per ticket, launched on `Discovered → Pending`, canonicalized in [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md); and (b) **phase subprocesses** — zero or more short-lived `claude -p '/kiro-* <args>' --output-format stream-json` invocations that A nominates via `action=run_phase`. The setup-judge and linear-updater subprocess shapes from the prior architecture are removed; the prior daemon-side mechanical kiro-spec / kiro-review gates are also removed (their work is absorbed by A as artifact validation per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md)).

#### Acceptance Criteria
1. When the orchestrator admits an issue (`Discovered → Pending`), the roki daemon shall launch orchestrator session A per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md): one `claude --input-format stream-json --output-format stream-json` subprocess in the issue's session tempdir, with `--settings` enforcing the `extension.orchestrator.allowed_tools` allowlist (Linear MCP write + `Read` + `Bash` by default), with the `prompt_template_orchestrator` block rendered as system prompt, and with a read-only filesystem sandbox (Bash invocations execute inside the sandbox so they cannot mutate the worktree).
2. While orchestrator session A is running, the roki daemon shall write JSON events to A's stdin per the event catalog in [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md) and shall parse A's stdout one turn at a time. The daemon shall extract the **last** JSON object on A's stdout per turn (after any extended-thinking block) and shall ignore reasoning text. The daemon shall validate the parsed object against A's response schema (see [fr:19-orchestrator-session > Response schema](../../../docs/fr/19-orchestrator-session.md)), including the `action`, `phase`, and `judge` enums.
3. If A emits no stdout for longer than a configurable orchestrator-stall window, the roki daemon shall SIGTERM A and route the issue to `Inactive(reason=orchestrator_crash)` per Requirement 12. The daemon shall not write a Linear-side notification on A's behalf in this case.
4. If A's response fails JSON schema validation on two consecutive turns (after one daemon-side reprompt), the roki daemon shall route the issue to `Inactive(reason=orchestrator_unparseable)` per Requirement 12 and shall capture A's raw stdout in the structured log.
5. The roki daemon shall enforce A's `max_phases` budget: each `action=run_phase` directive A emits consumes one slot of `extension.orchestrator.max_phases` (default `20`); when the budget is exhausted and A would nominate another phase, the daemon shall route the issue to `Inactive(reason=orchestrator_budget_exhausted)` per Requirement 12 and shall not spawn the additional phase. There is no per-turn budget on A; bounding is via `max_phases` and `effort`.
6. When A emits `action=run_phase` with `phase ∈ {materialize_spec, implement, validate, open_pr, ci_fix, finalize_review}` and `additional_context` (when present), the roki daemon shall spawn the corresponding phase subprocess: a single `claude -p '/<kiro-skill> <args>' --output-format stream-json` invocation in the issue's session tempdir for `materialize_spec` (kiro-discovery), `implement` (kiro-impl), `validate` (kiro-validate-impl), and `ci_fix` (kiro-debug + kiro-verify-completion), or with a daemon-internal small prompt fragment for `open_pr` (no skill) and `finalize_review` (review.md synthesis prompt, no skill). The daemon shall pass the configured per-phase `--max-turns` budget, render the per-phase context envelope (including A's `additional_context` verbatim through the engine adapter's `additional_context` channel per Requirement 13.4), and stream the phase subprocess's stdout as newline-delimited JSON events parsed into typed lifecycle events for structured logging.
7. If a phase subprocess emits no events for longer than a configurable per-phase stall window, the roki daemon shall terminate the subprocess, send a `phase_nonclean` event to A with the stall classification, and let A decide whether to re-run the phase, fall through to a `ci_fix` phase, or `action=stop`. If A is no longer alive when the phase stall is detected, the daemon shall route the issue to `Inactive(reason=stall)` and surface the failure via structured log + TUI escalation queue only.
8. When a phase subprocess clean-exits with terminal `result.subtype = success`, the roki daemon shall send a `phase_complete` event to A with the structured exit envelope and shall await A's next directive. When the phase clean-exits with a non-`success` `result.subtype`, exits non-zero without a terminal `result`, is signal-terminated, or exhausts its `--max-turns`, the daemon shall send a `phase_nonclean` event to A with the structured failure classification and shall await A's next directive — A may re-nominate the same phase, fall through to `ci_fix`, or `action=stop`.
9. When the terminal `result` event reports a `subtype` not present in the daemon's compiled mapping — including any new `subtype` Claude Code adds after the daemon's build — the roki daemon shall include the raw `subtype` value in the `phase_nonclean` event payload to A and capture it in the structured log; the daemon does not unilaterally route to `Inactive` for an unknown phase `subtype` because A is alive to make that judgment.
10. When the configured ticket-level retry budget for phase non-clean exits is exhausted (default three attempts, range one through ten), the roki daemon shall send a `daemon_directive` event with kind `retry_exhausted` to A so that A surfaces the failure to Linear and emits its own `action=stop` with `outcome=failure`. While remaining attempts exist the daemon shall transition the issue from `Active` to `Backoff`, apply exponential backoff between attempts bounded between ten seconds and five minutes, retain the session tempdir and the worktree across retries, and on timer expiry transition back to `Active` for A's next phase nomination.
11. When A emits `action=stop`, the roki daemon shall record the terminal `outcome` value (`success`, `failure`, or `cancelled`), publish the `Active → Inactive` transition (no subscriber veto — the prior review-gate vetoable hook is removed; structural validation of `review.md` is owned by A itself per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md)), and gracefully terminate A per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md). On `outcome=success` the issue lands in `Inactive(reason=awaiting_linear)`. On `outcome=failure` the daemon shall pick the originating `Inactive.reason` from the most recent failure observation already on the issue (e.g. `retry_exhausted` after a `daemon_directive (kind=retry_exhausted)`, or `awaiting_linear` if A simply chose to stop after a failed artifact-validation retry — the `reason` field is for cleanup eligibility and TUI surfacing, not for re-driving A). On `outcome=cancelled` the daemon shall route the issue through `Cleaning` per the `tracker_terminal` event path in [fr:04-state-machine-and-recovery](../../../docs/fr/04-state-machine-and-recovery.md).

### Requirement 6: Workspace-Level WORKFLOW.md Policy Loader

**Objective:** As an operator, I want a single workspace-level `WORKFLOW.md` file that defines orchestrator-session policy in Liquid + Markdown with schema validation and hot reload, so that I can adjust orchestrator behavior without restarting the daemon or recompiling Rust.

#### Acceptance Criteria
1. When the roki daemon starts, the WORKFLOW.md loader shall read the configured workspace-level `WORKFLOW.md`, parse its YAML or TOML front matter, identify the named template block `prompt_template_orchestrator` (consumed by orchestrator session A as the system prompt), and validate the result against the published schema. The previous `prompt_template_setup`, `prompt_template_worker`, and `prompt_template_linear_updater` named blocks are removed alongside the subprocess shapes they served.
2. If the workspace-level `WORKFLOW.md` fails schema validation at startup, the roki daemon shall refuse to start and shall log the validation error with the offending key path.
3. While the daemon is running, the WORKFLOW.md loader shall watch the workspace-level `WORKFLOW.md` for filesystem changes and shall re-validate the file before applying any changes. Hot-reload of the orchestrator namespace applies from the next ticket admission; an in-flight A keeps its rendered prompt for the lifetime of the session, per [fr:19-orchestrator-session > Configuration](../../../docs/fr/19-orchestrator-session.md).
4. If a hot-reload attempt produces a `WORKFLOW.md` that fails validation, the roki daemon shall keep the previously valid policy in effect and log the validation failure.
5. The WORKFLOW.md loader shall expose its parsed policy to the orchestrator through a stable schema shape that downstream specs can extend with additional keys without breaking existing consumers, and the loader shall round-trip unknown keys under the reserved extension namespaces (`extension.orchestrator.*`, `extension.server.*`) without interpreting them.
6. When the roki daemon launches orchestrator session A, the WORKFLOW.md loader shall render the `prompt_template_orchestrator` block against the active Linear issue with the issue's identifier, title, description, label list, and bucketed lifecycle state available as named variables; the rendered output shall become A's system prompt input.
7. Phase subprocess prompts are **not** templated by `WORKFLOW.md`. Each phase subprocess uses its skill's prompt as installed in the operator's local `~/.claude/skills/kiro-*` (`implement` → kiro-impl, `validate` → kiro-validate-impl, `ci_fix` → kiro-debug + kiro-verify-completion, `finalize_review` → review.md synthesis prompt) or, for `open_pr` (no skill) and `finalize_review` (synthesis), a small daemon-internal prompt fragment. The daemon shall pass any per-phase `additional_context` from A through the engine adapter's stable forwarding channel (Requirement 13.4) regardless.
8. If the `prompt_template_orchestrator` block fails to render at A launch, the roki daemon shall log the render failure with the offending block name and shall provide a deterministic fallback prompt that includes the Linear issue identifier, title, and description so that A always receives non-empty issue context.

### Requirement 7: Agent Tooling Boundary

**Objective:** As an operator, I want the daemon to neither register nor proxy any agent-side tool, so that every agent invocation (orchestrator session A, every phase subprocess) inherits exactly the tool surface my local Claude Code installation exposes (Bash plus my installed MCP servers), narrowed only by per-process `allowed_tools`, and the daemon never needs to hold Linear or git write credentials on the agent's behalf.

#### Acceptance Criteria
1. The roki daemon shall not register, proxy, or wrap any agent-side tool inside orchestrator session A or any phase subprocess; each subprocess's tool surface shall be exactly what the operator's local Claude Code installation exposes — its built-in tool set plus any MCP servers the operator has installed — narrowed only by per-process `allowed_tools` lists the daemon assembles from configuration. For A specifically the surface is Linear MCP (write) + `Read` per [fr:19-orchestrator-session > Tool surface](../../../docs/fr/19-orchestrator-session.md).
2. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from orchestrator session A (through the operator-configured Linear MCP integration) or from a phase subprocess where its `allowed_tools` permits, and the daemon shall not hold any agent-facing Linear write path.
3. Where the workspace-level `WORKFLOW.md`, A's rendered prompt, or a phase subprocess's installed skill prompt instructs an agent invocation to operate `git`, `gh`, `ghq`, `wt`, or Linear tooling from inside the subprocess, the operator's local Claude Code installation shall be the source of those affordances (Bash for shell tools and the operator's installed MCP servers for Linear); the roki daemon shall not substitute, intercept, or augment those tool invocations.
4. The roki daemon shall never embed the Linear API token or the Linear webhook secret in any prompt input, log line, or other artifact reachable from inside any agent subprocess.

### Requirement 8: Orchestrator State Machine and Extension Points

**Objective:** As a downstream spec author, I want the orchestrator to expose stable per-issue state-machine hooks, so that roki-spec-gate, roki-review-gate, and roki-observability can subscribe to lifecycle transitions without forking the core.

#### Acceptance Criteria
1. The roki daemon shall maintain an in-memory state machine per Linear issue identifier covering exactly the following six daemon-local states: `Pending`, `Judging`, `Active`, `Backoff`, `Inactive`, and `Cleaning`. These states represent the daemon's bookkeeping for subprocess lifecycle and cleanup; the daemon does not mirror Linear's workflow states (review, done, etc.), which it observes through the tracker rather than maintaining internally. `Inactive` shall carry a `reason` discriminator drawn from the set `{noop, awaiting_linear, needs_split, allowlist_rejected, orchestrator_crash, orchestrator_unparseable, orchestrator_budget_exhausted, stall, retry_exhausted, fs_poison, orphan}`, used for structured logging, TUI surfacing, and `Cleaning` eligibility but not for orchestrator-internal transition decisions. `Pending` is the staging state covering admission and any non-running idle waiting (it is the entry state from `Discovered` admission and from re-admission of an `Inactive` entry per Requirement 3.14); orchestrator session A is launched on `Discovered → Pending` per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md). `Judging` is the state during which A is processing an `admission_request` event. `Backoff` is the retry-window state for ticket-level retries between phase subprocess non-clean exits; on timer expiry it returns to `Active` for A's next phase nomination. `Cleaning` shall be reachable only from `Active`, `Backoff`, or `Inactive`, and only as a result of a tracker observation (the issue moved to a configured terminal Linear state, or the issue was reassigned away from the configured assignee); a phase subprocess exit alone shall never trigger entry to `Cleaning`. The state machine shall declare no vetoable transitions: the prior `Judging → Active` (spec gate) and `Active → Inactive` (review gate) hooks are removed alongside the gates themselves; structural validation of `requirements.md` and `review.md` is owned by A inside its own phase-planning loop per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md).
2. When a state transition occurs, the roki daemon shall publish a structured transition event whose payload identifies the previous state, the next state, the trigger source, and the issue identifier, and shall include the originating repository identifier when one applies. When the next state is `Inactive`, the payload shall additionally include the `reason` discriminator.
3. The roki daemon shall expose subscription hooks that allow other components to observe transition events read-only; no transition is vetoable (the prior `Judging → Active` and `Active → Inactive` veto windows are removed alongside the gates per 8.1).
4. If a subscriber raises an unhandled error while processing a transition event, the roki daemon shall isolate the failure to that subscriber, log the error with the subscriber's identifier, and continue processing transitions for other subscribers.
5. The roki daemon shall recover the per-issue state on restart by re-reading Linear and the filesystem layout, without depending on a persistent database. Orchestrator session A is not persisted across daemon restarts: a fresh A is launched when an issue re-enters `Pending` per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md).

### Requirement 9: Permission Strategy and Default Sandbox

**Objective:** As an operator, I want roki to run agents under the safest workable permission profile while letting me opt into a fallback when Claude's allowlist is unreliable, so that I can balance safety against operability.

#### Acceptance Criteria
1. The roki daemon shall launch each phase subprocess with the agent sandbox set to `workspace-write` and with elicitations rejected by default.
2. When the workspace-level `WORKFLOW.md` declares an alternative sandbox or elicitation policy, the roki daemon shall apply that override to every phase subprocess.
3. When the operator selects the `--settings` allowlist permission strategy, the roki daemon shall pass the configured allowlist to each phase subprocess through Claude Code's settings interface.
4. When the operator selects the `--dangerously-skip-permissions` fallback strategy through configuration or the `--dangerously-skip-permissions` CLI flag, the roki daemon shall pass that flag to each phase subprocess and shall log the elevated-permission decision per phase launch. The fallback strategy shall not apply to orchestrator session A; A's tool surface is governed exclusively by `extension.orchestrator.allowed_tools` via `--settings`.
5. If neither permission strategy is configured for phase subprocesses, the roki daemon shall refuse to start and shall report the missing configuration.
6. The roki daemon shall launch orchestrator session A with a read-only filesystem sandbox and rejected elicitations regardless of operator overrides per [fr:19-orchestrator-session > Tool surface](../../../docs/fr/19-orchestrator-session.md). A's `Bash` capability (when permitted by `extension.orchestrator.allowed_tools`) executes inside that sandbox so commands can read state but cannot mutate the worktree or session tempdir. Whether A can call write-capable Linear MCP tools is governed by the operator's Claude Code MCP allowlist intersected with `extension.orchestrator.allowed_tools`, not by the daemon's permission strategy.

### Requirement 10: Restart Recovery Without Persistent Storage

**Objective:** As an operator, I want roki to recover its working state after a restart by re-reading Linear and the filesystem rather than relying on a database, so that I never carry stale or corrupted local state across restarts.

#### Acceptance Criteria
1. When the roki daemon starts, it shall list every session tempdir under the platform-appropriate user cache root and, for every configured allowlisted repository, list every existing git worktree whose branch name matches the operator-configurable Linear issue-identifier pattern.
2. The roki daemon shall reconcile every distinct issue identifier discovered from session tempdirs and from worktree branches against Linear before resuming work, shall apply the configured Linear assignee filter during reconciliation, and shall classify each discovered issue into one of the documented recovery states (resume-active, orphaned-session, orphaned-worktree, fresh-queued, no-op), mapping resume-active to `Pending` (which then progresses through the normal admission flow with a freshly launched orchestrator session A) and the orphan classifications to `Inactive(reason=orphan)`.
3. If a session tempdir or worktree exists for an issue identifier that has no matching active Linear issue assigned to the configured Linear assignee at startup, the roki daemon shall mark that residue as `Inactive(reason=orphan)`, log it without deleting it automatically, and — if and when a future orchestrator session A is launched for any in-flight issue — surface the orphan via a `daemon_directive` event with kind `orphan` so that A writes the appropriate Linear label + comment. If no in-flight A exists, the orphan surfaces via structured log + TUI escalation queue only, per Requirement 12.
4. If a Linear issue is assigned to the configured Linear assignee and is in a Linear workflow state contained in the configured `admit_states` set at startup, but has no matching session tempdir or worktree, the roki daemon shall enqueue the issue for a fresh admission cycle starting from `Pending` (which launches a fresh orchestrator session A).
5. The roki daemon shall not write any per-issue runtime state to disk except the session tempdir and worktree contents the agent itself produces and the structured logs the daemon emits.

### Requirement 11: Observability of Daemon Internals

**Objective:** As an operator debugging a stuck ticket, I want enough structured observability from the daemon itself to diagnose orchestrator, phase, session, worktree, and tracker issues without an external UI, so that the MVP is operable before the dedicated observability spec ships.

#### Acceptance Criteria
1. The roki daemon shall emit a structured log event for every orchestrator session A lifecycle change, every phase subprocess lifecycle change, every session tempdir creation or deletion, every worktree creation or removal, every Linear poll or webhook receipt, every backoff or stall decision, every retry attempt with its attempt counter, every `daemon_directive` event sent to A, every A response (action + judge + phase + outcome fields, redacted of `reason` when long), and every state-machine transition (including the `Inactive` `reason` discriminator). The canonical event catalog lives in [`docs/reference/log-events.md`](../../../docs/reference/log-events.md).
2. The roki daemon shall include the Linear issue identifier when one applies, the repository identifier when one applies (for example for worktree-scoped events), and a correlation identifier for the originating subprocess invocation in every structured log event.
3. The roki daemon shall support a configurable log level and a configurable log destination so that the operator can route logs to stdout, a file, or both.
4. The roki daemon shall redact the Linear API token, the Linear webhook secret, and any other operator-declared secrets from every structured log event before emission.
5. While orchestrator session A or any phase subprocess is running, the roki daemon shall capture every line emitted on the subprocess's standard error stream and shall surface each non-empty line as a structured log event at warn severity tagged with the issue identifier, the subprocess role (`orchestrator` or `phase:<phase-name>`), and the correlation identifier.
6. Where the operator opts into per-issue debug capture through the documented CLI flag or configuration block, the roki daemon shall append every line emitted on each subprocess's standard output and standard error streams to a per-issue file under the configured debug-log directory, and each appended line shall carry an RFC 3339 timestamp with nanosecond resolution and a stream tag identifying standard output or standard error.
7. If the per-issue debug log file cannot be opened or appended to, the roki daemon shall log the failure with the offending file path and shall continue running the subprocess without aborting the launch.
8. When orchestrator session A or any phase subprocess completes (success, retry, or final failure), the roki daemon shall emit a structured log event recording the subprocess role, the duration, the parsed outcome (when parseable), and the issue identifier.

### Requirement 12: Daemon-Only Failure Surfacing

**Objective:** As an operator, I want roki to surface daemon-only failures that no agent has yet self-reported — into Linear (when orchestrator session A is alive to do it on the daemon's behalf) and into the optional TUI escalation queue (always) — so that I learn about stuck or failed tickets without continuously tailing daemon logs and without holding any Linear write path inside the daemon process.

The canonical surfacing channel is now **A processing a `daemon_directive` event and writing Linear via Linear MCP**. When A is dead — because of `orchestrator_crash`, `orchestrator_unparseable`, or `orchestrator_budget_exhausted` — the daemon does **not** fall back to a Linear write of its own; the failure surfaces via structured log + TUI escalation queue only, per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md).

#### Acceptance Criteria
1. The roki daemon shall maintain an in-memory escalation queue keyed by Linear issue identifier that records the most recent daemon-only failure event per issue (failure category, structured fields, timestamp, correlation identifier when applicable, repository identifier when applicable). The escalation queue is the data source consumed by the optional `OrchestratorRead` snapshot exposed to roki-observability.
2. When the daemon detects a daemon-only failure event listed below **and orchestrator session A is alive for the affected issue**, the roki daemon shall send a `daemon_directive` event to A's stdin per [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md). A then writes the appropriate Linear label + comment via Linear MCP and returns `action=linear_update_done`. The `daemon_directive` covers:
   (a) phase stall after stall detection (Requirement 5.7) — the daemon killed the phase subprocess so the phase agent had no chance to self-report;
   (b) retry budget exhaustion after phase non-clean exits (Requirement 5.10);
   (c) filesystem poison during session or worktree creation, removal, or rename (Requirement 4.9);
   (d) startup-recovery orphan retained for inspection (Requirement 10.3);
   (e) any other daemon-detected failure where A is alive and Linear-side surfacing is needed.
   The daemon does **not** send a `daemon_directive` for the `needs_split` and `allowlist_rejected` admission rejections — A writes those Linear updates itself in the same turn it returns the `admission_decision`, per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). Artifact-validation retry-budget exhaustion (`requirements.md` after `materialize_spec`, `review.md` after `finalize_review`) is owned by A entirely — A writes the Linear feedback itself and emits `action=stop` with `outcome=failure`; the daemon does not send a directive for it.
3. When the daemon detects a failure that kills A itself, the roki daemon shall route the issue to one of three `Inactive(reason=*)` values per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md) and shall **not** attempt a Linear write on A's behalf:
   (a) A process crash, SIGSEGV, non-zero exit without a `stop` action, or A stall (no event emitted in N seconds, configurable) → `Inactive(reason=orchestrator_crash)`;
   (b) A schema drift on two consecutive turns after one daemon-side reprompt → `Inactive(reason=orchestrator_unparseable)`;
   (c) `max_phases` exhausted (A would nominate another phase but the budget is gone) → `Inactive(reason=orchestrator_budget_exhausted)`.
   In all three cases the daemon shall log the failure structurally, populate the escalation queue (Requirement 12.1), and surface the issue exclusively via the TUI escalation queue from roki-observability — there is no Linear-side notification. These three reasons are **not** auto-cleanup eligible: the worktree and session tempdir are preserved until the operator manually closes the Linear ticket, after which `Cleaning` proceeds.
4. The roki daemon shall not send `daemon_directive` events for events that an agent (A or a phase subprocess) is expected to self-report through Linear, including normal phase completions, agent-recoverable errors the agent surfaces in the ticket itself, and the review-gate fix-finding loop's intentional re-launches.
5. Each `daemon_directive` event shall pass at minimum the Linear issue identifier, the daemon-only failure `kind`, the structured fields associated with the failure (e.g. `correlation_id`, `repos[]`, `worktree_path`, `last_subtype`, `attempts`, `window_ms`, `errno`), and the event timestamp; the directive shall not include the Linear API token, the Linear webhook secret, or any other operator-declared secret.
6. The actual Linear label name(s) and comment text A applies are determined by A's `prompt_template_orchestrator` system prompt and any operator instructions therein; the daemon contributes only the directive `kind` and the structured fields and does not template the Linear write itself.
7. If A's response to a `daemon_directive` indicates a partial Linear write (the `linear_writes` field in `action=linear_update_done` shows a subset of the expected writes), the roki daemon shall log the partial-write entry, retain the escalation queue entry, and shall not retry on A's behalf. If A fails entirely on a `daemon_directive` (turn ends with an error, or A crashes mid-turn), the daemon shall log the failure and route the issue to `Inactive(reason=orchestrator_crash)` per Requirement 12.3 — the escalation queue entry stays so the TUI continues to surface it.

### Requirement 13: Cross-Spec Extension Surface

**Objective:** As a downstream spec author, I want roki-mvp to publish a stable, additive extension surface, so that roki-spec-gate, roki-review-gate, and roki-observability can integrate without forking the core or breaking sibling specs.

#### Acceptance Criteria
1. The roki daemon shall publish a read-only `OrchestratorRead` trait that exposes a snapshot of the per-issue state (including the `Inactive` `reason` discriminator with the three orchestrator-dead values from Requirement 12.3) plus a single-issue lookup, plus a snapshot of the escalation queue from Requirement 12.1, and shall grant no state-mutation rights through that trait.
2. The roki daemon shall declare no vetoable state transitions; the prior `Judging → Active` (spec gate) and `Active → Inactive` (review gate) hooks are removed alongside the gates. Structural validation of `requirements.md` and `review.md` is owned by orchestrator session A inside its own phase-planning loop per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md), not by a daemon-side subscriber.
3. The roki tracker adapter shall publish a `TrackerRefresh` nudge trait that allows external callers to request an out-of-cycle poll without bypassing the documented cadence cap or the 429 backoff state.
4. The roki engine adapter shall accept an additive optional `additional_context` field on the per-phase context and shall forward its value verbatim to the phase subprocess through a stable, machine-extractable section of the phase subprocess's prompt envelope, kept distinct from the skill's installed prompt body, without the daemon interpreting the contents. The exact serialization shape (delimiter sentinels, JSON envelope, named keys) is defined by the engine adapter design and is treated as additive: new optional keys may be added under the same forwarding contract; removing or retyping existing keys is a breaking change. A populates `additional_context` directly on `action=run_phase` directives — including the artifact-validation retry path (e.g. failing per-criterion entries A read from `review.md` injected into the next `implement` phase) per [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md).
5. The workspace-level `WORKFLOW.md` schema shall reserve the `extension.orchestrator.*` and `extension.server.*` sub-namespaces for the canonical roki specs and orchestrator session A; the loader shall round-trip unknown keys under those namespaces without interpreting them. The previously reserved `extension.linear_updater.*`, `extension.gates.spec.*`, and `extension.gates.review.*` namespaces are removed alongside the subprocess shapes / gates they served.
