# Requirements Document

## Project Description (Input)
Developers manually shepherd Linear tickets through implementation: read ticket, set up repo, prompt the agent, watch it work, transition Linear states, open PR. The supervision burden scales linearly with ticket volume, and humans drift between tickets while waiting on the agent. roki-mvp delivers a near-one-way Linear -> PR path with guardrails that doesn't require constant minding. It is the symphony-parity vertical slice: a `roki` Rust binary running as a daemon that polls Linear (or accepts webhooks). Each incoming Linear event passes a daemon-side mechanical pre-admission-judge in Rust (assignee match + Linear state in `admit_states` + fixed labels `roki:ready` and optionally `roki:impl`); a ticket that fails any check is silently skipped (log only, no state entry, no Linear write). A passing ticket enters `Pending` with a per-ticket `mode` flag — `SPEC_DRIVEN` when `roki:impl` is also present (operator declared the project-level `<repo>/.kiro/specs/<target>/` is complete), `NEEDS_CLASSIFY` when only `roki:ready` is present — and the daemon launches a long-lived **orchestrator session**: a `claude --input-format stream-json --output-format stream-json` "thinking" component whose tool surface is restricted to the operator's installed Linear MCP (write) plus `Read` and `Bash` (read-only filesystem sandbox) and whose responses are strict JSON action directives over a small enum. The orchestrator absorbs target-spec resolution (SPEC_DRIVEN), classify-driven path branching (NEEDS_CLASSIFY), phase planning, structural artifact validation (`review.md` after `finalize_review`; SPEC_DRIVEN target spec docs on the first turn), daemon-only failure surfacing, and Linear writes; the daemon never writes Linear directly and never decides whether an artifact passes. For each phase the orchestrator nominates, the daemon spawns a short-lived bounded phase subprocess that performs the actual code-changing work; the catalog default for each phase is either a slash-command-driven skill (`claude -p '/<skill> <args>' --output-format stream-json`) or a daemon-internal Liquid prompt template rendered onto stdin, and operators may override per phase via `WORKFLOW.md`. Phase catalog: `classify` (NEEDS_CLASSIFY first turn only; default `roki-classify`, returns Path A/B/C/D/E plus a recommended next operator command and label), `implement` (default depends on mode: SPEC_DRIVEN uses `kiro-impl` against the project-level spec, NEEDS_CLASSIFY Path B uses the daemon-internal prompt `prompt_template_implement_direct` against the ticket body's `## Acceptance Criteria`), `review` (default `kiro-review`, feature-level adversarial code-quality pass with mode-dependent criteria source), `validate` (default depends on mode: SPEC_DRIVEN uses `kiro-validate-impl`, NEEDS_CLASSIFY uses `prompt_template_validate_direct` for the same two-stage mechanical/acceptance check against the ticket body), `open_pr` (daemon-internal prompt `prompt_template_open_pr`, no default skill), `ci_fix` (default `roki-ci-fix`, fetches CI logs via `gh` and categorizes failures), `finalize_review` (default `roki-finalize-review`, synthesizes `review.md` per the schema in the artifacts reference; criterion ID source is mode-dependent — SPEC_DRIVEN uses `requirements.md` numeric IDs, NEEDS_CLASSIFY uses ticket body EARS numbers). The previous per-issue `materialize_spec` phase is removed: SPEC_DRIVEN reuses the operator-completed project-level spec, NEEDS_CLASSIFY's `classify` decides whether the work is small enough for direct implementation (Path B) or needs operator handoff (Path A/C/D/E). The daemon configures an allowlist of repos. The orchestrator either resolves the target repo from the project-level spec directory (SPEC_DRIVEN) or from the `classify` phase's Path-B context (NEEDS_CLASSIFY) and validates allowlist membership itself; multi-repo or out-of-allowlist tickets stop with the matching `outcome` and an orchestrator-driven Linear comment. The daemon then materializes the corresponding git worktree via `wt` + `ghq`. The agent inside every phase subprocess inherits the operator's local Claude Code tool surface (Bash plus the operator's installed MCP servers, including Linear MCP); the daemon never registers, proxies, or wraps agent-side tools. Per-issue state is keyed by Linear issue id alone. `WORKFLOW.md` (Liquid + Markdown, hot reload) is a single workspace-level policy artifact whose required named template blocks are `prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, and `prompt_template_open_pr`.

## Introduction

The roki-mvp specification defines the foundational vertical slice of the roki system: a single Rust daemon that watches Linear for active tickets, gates them through a mechanical pre-admission-judge, allocates per-issue sessions, and supervises two shapes of bounded Claude Code subprocess invocations that perform the actual implementation work. The daemon is a passive observer of Linear state and an active controller of subprocess lifecycle, session filesystem state, and on-demand git worktree provisioning — it never mutates Linear, never opens pull requests, and never edits code. All write effects on Linear, GitHub, and the working tree are delegated to agents running inside subprocess sandboxes; the daemon's view into each agent loop is a thin observation layer over stream-json output, used for structured logging, stall detection, schema-drift detection, and final-outcome classification rather than for driving the agent.

This MVP is symphony-aligned: in-memory orchestrator with no persistent database, recovery driven by re-reading Linear and the filesystem on restart, a single workspace-level `WORKFLOW.md` as the user-facing policy boundary, and a two-shape engine taxonomy per admitted issue. The two shapes are: (a) one long-lived **orchestrator session** (`claude --input-format stream-json --output-format stream-json`) that emits strict JSON action directives, absorbing target-spec resolution and classify-driven path branching (replacing the prior setup-judge subprocess), the daemon-only failure surfacing channel (replacing the prior linear-updater subagent), and artifact validation (replacing the prior daemon-side mechanical kiro-spec / kiro-review gates); and (b) zero or more short-lived bounded **phase subprocesses** (`claude -p '/<skill> <args>'` for skill-driven phases or `claude --input-format stream-json` with daemon-internal Liquid template for direct-mode and `open_pr`) that the orchestrator nominates via `run_phase`. roki diverges from symphony in being daemon-driven on workspace lifecycle (the daemon — not the orchestrator — materializes and reclaims the worktree, both via `wt` + `ghq` against the operator's allowlist), in applying a daemon-side mechanical pre-admission-judge before the orchestrator is launched (so tickets that fail assignee / state / label gates cost zero subprocess), in inheriting the operator's local Claude Code tool surface inside each phase subprocess (no daemon-registered or daemon-proxied agent tools), and in publishing stable extension points so dependent specs (roki-observability) can plug in without forking the orchestrator.

The orchestrator's full lifecycle, event catalog, response schema, tool surface, configuration, artifact-validation contract, and failure modes are canonicalized in [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). The phase catalog (per-mode defaults, override forms, exit envelope) is canonicalized in [fr:18-worker-skill-workflow](../../../docs/fr/18-worker-skill-workflow.md). The pre-admission-judge and the per-issue state machine (five states + an `Inactive.reason` discriminator) are canonicalized in [fr:04-state-machine-and-recovery](../../../docs/fr/04-state-machine-and-recovery.md). This requirements document references those contracts rather than restating them.

## Boundary Context

- **In scope**: Rust binary with CLI, async runtime, structured logging; single workspace-level `WORKFLOW.md` loader with Liquid templating, Markdown front matter, schema validation, and hot reload, exposing four required named template blocks (`prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, `prompt_template_open_pr`); in-memory state machine for per-issue lifecycle keyed by Linear issue id (five states — `Pending`, `Active`, `Backoff`, `Inactive`, `Cleaning` — with an `Inactive` reason discriminator that includes `awaiting_linear`, `needs_operator`, `spec_incomplete`, `needs_split`, `allowlist_rejected`, `orchestrator_crash`, `orchestrator_unparseable`, `orchestrator_budget_exhausted`, `stall`, `retry_exhausted`, `fs_poison`, `orphan`); Linear GraphQL client (read-only on the daemon side) with a single workspace-level webhook receiver and a single workspace-level polling fallback; daemon-side mechanical pre-admission-judge in Rust evaluating each Linear webhook against four ordered conditions (assignee match, Linear state in `admit_states`, `roki:ready` label present, `roki:impl` label optionally present to select `mode=SPEC_DRIVEN` vs `mode=NEEDS_CLASSIFY`); fixed Linear label conventions (`roki:ready` and `roki:impl`, not operator-configurable); tracker normalization (issue model, state extraction, label extraction, assignee extraction); long-lived orchestrator session per admitted issue (one `claude --input-format stream-json --output-format stream-json` invocation, launched on entry to `Pending` after the pre-admission-judge passes, configured under the reserved `extension.orchestrator.{model, effort, max_phases, allowed_tools, stall_seconds}` namespace in `WORKFLOW.md`, runs with a read-only filesystem sandbox regardless of operator overrides, with the per-ticket `mode` flag substituted into the rendered system prompt); orchestrator-driven first-turn behavior depending on mode — SPEC_DRIVEN resolves the target spec name from the ticket body via LLM inference and structurally validates `<repo>/.kiro/specs/<target>/{spec.json,requirements.md,design.md,tasks.md}` using `Read` + `Bash`, NEEDS_CLASSIFY nominates the `classify` phase as the first `run_phase` directive; daemon-driven idempotent worktree ensure (via `ghq list -p` + `wt switch-create` on the first call, then `wt list` verification on subsequent calls) on every non-classify phase nomination for a validated single allowlisted repo (the `classify` phase MUST NOT receive a worktree); per-issue session tempdir lifecycle (created on entry to `Pending`, ephemeral per issue); orchestrator-driven `review.md` validation (after each `finalize_review` clean exit the orchestrator reads `review.md` and decides `action=stop` or retry, using `Read` + `Bash` only inside its read-only filesystem sandbox); short-lived bounded phase subprocesses for code-changing work — one phase subprocess per phase that the orchestrator nominates, each with its own `--max-turns` budget, either a slash-command-driven skill or a daemon-internal Liquid template, stream-json parsing, and stall detection; phase catalog with operator-overridable defaults and mode-aware selection — `classify` (NEEDS_CLASSIFY first turn only; default `roki-classify`, max_turns 5), `implement` (SPEC_DRIVEN default `kiro-impl`, NEEDS_CLASSIFY default daemon-internal `prompt_template_implement_direct`), `review` (default `kiro-review`, feature-level adversarial code-quality pass with mode-dependent criteria source), `validate` (SPEC_DRIVEN default `kiro-validate-impl`, NEEDS_CLASSIFY default daemon-internal `prompt_template_validate_direct`), `open_pr` (daemon-internal `prompt_template_open_pr`, no default skill), `ci_fix` (default `roki-ci-fix`), `finalize_review` (default `roki-finalize-review`, criterion ID source is mode-dependent); each phase's invocation may be overridden via `WORKFLOW.md` using `extension.phase.<name>.command` (slash-command swap) or a `prompt_template_<phase>` named template block (templated stdin), mutually exclusive per phase; the orchestrator processes `daemon_directive` events for daemon-only failure surfacing (stall, retry exhaustion, fs poison, orphan recovery, etc.) and writes labels + comments to Linear via the operator's installed Linear MCP within its own session — the daemon never issues a Linear write itself; orchestrator-driven operator-facing pre-phase stops via `action=stop` with `outcome ∈ {needs_split, allowlist_rejected, spec_incomplete, needs_operator}` (the orchestrator writes the matching Linear feedback in the same turn); operator-declared allowlist of repos that bounds which repositories the daemon will materialize worktrees in (the allowlist is the daemon's enforcement boundary for the orchestrator's repo resolution; multi-repo or out-of-allowlist tickets are rejected by the orchestrator emitting `outcome=needs_split` or `outcome=allowlist_rejected`); daemon-driven worktree cleanup via allowlist iteration plus `wt remove` for any branch whose name equals the Linear issue identifier verbatim; bounded loops (per-phase `--max-turns` flag passthrough enforced by the phase subprocess; `max_phases` budget enforced by the daemon for the orchestrator's phase nominations, default `15`; exponential backoff between ticket-level retry attempts bounded between ten seconds and five minutes on phase non-clean exit; configurable retry budget on phase non-clean exit); configurable permission strategy (`--settings` allowlist with `--dangerously-skip-permissions` fallback) for phase subprocesses; default phase-subprocess sandbox set to `workspace-write` with elicitations rejected, overridable through the workspace-level `WORKFLOW.md`; the orchestrator always runs with a read-only filesystem sandbox regardless of operator overrides, with `Bash` permitted but unable to mutate the filesystem due to that sandbox.
- **Out of scope**: any logic that writes Linear state, opens or comments on pull requests, or edits source files — Linear writes originate exclusively from the orchestrator through the operator's installed Linear MCP, and code / PR effects originate exclusively from phase subprocesses through the operator's local Claude Code tool surface; daemon-registered, daemon-proxied, or daemon-wrapped agent-side tools (every subprocess inherits the operator's Claude Code installation as-is, narrowed only by per-process `allowed_tools`); persistent state stores (SQLite or otherwise); per-issue spec materialization inside the daemon flow (the previous `materialize_spec` phase is removed; SPEC_DRIVEN reuses the operator-authored project-level spec, NEEDS_CLASSIFY hands off to the operator for spec authoring on Path A/C/D/E); daemon-side mechanical artifact-validation gates (the prior roki-spec-gate / roki-review-gate vetoable hooks are removed; the orchestrator reads the SPEC_DRIVEN target spec docs and `review.md` itself per its own retry-budget rules); HTTP API and TUI observability surfaces (deferred to roki-observability); post-merge flow-document distill sweep (handled in CI, not by the daemon); container or VM isolation; multi-host SSH workers; auto-merge orchestration; per-repo `WORKFLOW.md` overrides; multi-repo tickets (one ticket = one repo; multi-repo classifications are rejected by the orchestrator via `action=stop` with `outcome=needs_split`); operator-renamable label conventions (`roki:ready` and `roki:impl` are fixed strings); mode mutation mid-flight (relabeling a ticket while its orchestrator is running does not re-route it; the next webhook re-runs the pre-admission-judge); Slack and other push-style operator notification channels (daemon-only failures surface via the orchestrator → Linear MCP when the orchestrator is alive, and via TUI escalation queue + structured log only for the three orchestrator-dead failure paths and for skipped tickets); orchestrator-side context compaction across long-running tickets (deferred — the `max_phases` budget bounds session length for MVP); Rust-native read-only artifact-validation tool to replace the orchestrator's `Bash` use (deferred — for MVP the orchestrator uses Bash directly); Windows support.
- **Adjacent expectations**: the operator installs Claude Code locally with kiro and roki skills available as personal skills under `~/.claude/skills/kiro-*` and `~/.claude/skills/roki-*` (not vendored, not plugin-namespaced); the operator installs the `wt` (worktrunk) and `ghq` external CLIs and ensures both are on `$PATH`; the operator installs and configures a Linear MCP integration (or equivalent agent-side Linear tooling) inside their Claude Code installation, so that the orchestrator and phase subprocesses can read and write Linear without the daemon proxying any Linear write path; the operator provides a Linear API token (used by the daemon for read-only tracker access only) and a single workspace-level Linear webhook secret; the operator declares the Linear assignee in the daemon configuration file (`roki.toml` by default), using `me` when roki should resolve the operator from the Linear API token; the operator applies the fixed labels `roki:ready` (always required for daemon admission) and `roki:impl` (when declaring an existing project-level spec is complete) to Linear tickets; the operator configures one or more repository allowlist entries by their `ghq` identifier; the operator authors a single workspace-level `WORKFLOW.md` with the four required named template blocks (`prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, `prompt_template_open_pr`) or relies on the bundled defaults; downstream spec (roki-observability) depends on stable state-machine hooks, the orchestrator's response schema, and the workspace-level `WORKFLOW.md` schema published by this MVP.

## Requirements

### Requirement 1: Daemon Lifecycle and CLI

**Objective:** As an operator, I want a single `roki` binary that runs as a long-running daemon and exposes a clear CLI, so that I can start, configure, and stop the system without bespoke scripting.

#### Acceptance Criteria
1. When the operator invokes `roki run` with a valid configuration, the roki daemon shall start the orchestrator runtime, the Linear adapter, the workflow loader, and the webhook server before reporting ready.
2. If the configuration file is missing, malformed, or fails schema validation, the roki daemon shall exit with a non-zero status and emit a structured log entry that names the offending field.
3. If the `wt` external CLI, the `ghq` external CLI, or the configured `claude` binary is not present at startup, the roki daemon shall refuse to start and shall log an actionable remediation message that names the missing executable.
4. When the operator sends SIGINT or SIGTERM to a running daemon, the roki daemon shall stop accepting new work, signal each active orchestrator session and each active phase subprocess to terminate, await a bounded shutdown window per subprocess, and exit cleanly.
5. The roki daemon shall emit structured logs through a tracing pipeline that records per-issue and per-subprocess context fields for every event it produces, plus a per-repo field for events scoped to a specific repo (for example worktree creation outcomes and worktree cleanup). The structured event catalog is canonical in [`docs/reference/log-events.md`](../../../docs/reference/log-events.md).
6. The roki daemon shall accept the CLI flags `--config <path>`, `--bind <addr>`, `--port <num>`, `--dangerously-skip-permissions`, and `--debug`, where flags override configuration values; the daemon shall document each flag in `--help` output for `roki run` and any subcommand. The canonical CLI surface is [`docs/reference/cli.md`](../../../docs/reference/cli.md).
7. When the operator invokes `roki --help` or any subcommand with `--help`, the roki daemon shall print usage information that documents all configuration knobs surfaced through the CLI.

### Requirement 2: Configuration and Pre-Admission Judge

**Objective:** As an operator, I want to configure one daemon to serve only my assigned Linear tickets across an allowlist of Git repositories with shared Linear credentials, and I want the daemon to gate every incoming Linear event through a mechanical pre-admission filter (assignee + Linear state + fixed labels) before launching any subprocess, so that I can run a single roki instance across my whole project portfolio without consuming tickets I have not authorized.

#### Acceptance Criteria
1. The roki daemon shall accept a configuration source that declares zero or more repository allowlist entries, where each entry is identified by its `ghq` repository identifier (`owner/repo` or `host/owner/repo`) and where the local clone path is resolved at runtime through `ghq`.
2. If two configured repository allowlist entries declare the same `ghq` identifier, the roki daemon shall refuse to start and shall log the offending duplicate entry.
3. The roki daemon shall accept a single workspace-level `[linear]` configuration block declaring the Linear API token source and the Linear webhook secret source, and shall refuse to start if the Linear API token or the webhook secret cannot be resolved.
4. The roki daemon shall accept a single workspace-level `[workflow]` configuration block declaring the path to the workspace-level `WORKFLOW.md`, and shall refuse to start if that path is missing or unreadable.
5. The roki daemon shall accept a `[server]` configuration block declaring the bind address and port for the webhook receiver, and shall apply CLI overrides for both fields when supplied.
6. The roki daemon shall key all per-issue runtime state by the Linear issue identifier alone, independently of any specific repository.
7. If the configured repository allowlist is empty, the roki daemon shall start with a warning, and any subsequent orchestrator-driven repo resolution shall fail allowlist validation — affected admitted issues effectively flow to `Inactive(reason=allowlist_rejected)` until the operator configures at least one repository.
8. The roki daemon shall accept a single workspace-level Linear assignee value in the daemon configuration file (`roki.toml` by default) under the `[linear]` configuration block, where the value `me` means the Linear user associated with the configured Linear API token.
9. If the Linear assignee is missing, empty, or cannot be resolved to exactly one Linear user, the roki daemon shall refuse to start and shall log the offending configuration field.
10. The roki daemon shall accept a workspace-level `[linear].admit_states` configuration knob declaring the set of Linear workflow state names eligible for admission (default: `["Todo"]`); the daemon shall refuse to start if the resolved set is empty.
11. The roki daemon shall reserve the `extension.orchestrator.*` configuration namespace under the workspace-level `WORKFLOW.md` for the orchestrator session; the namespace shall accept the keys `model`, `effort`, `max_phases`, `allowed_tools`, and `stall_seconds` per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). When any of these keys is omitted, the daemon shall apply the canonical defaults documented in that FR (`model = "claude-opus-4-7"`, `effort = "middle"`, `max_phases = 15`, `allowed_tools` permitting Linear MCP write + `Read` + `Bash`, `stall_seconds = 120`). If any reserved key under that namespace fails type, enum, or range validation at startup or hot reload, the daemon shall refuse the new policy at startup or retain the previous policy at hot reload, and shall log the offending key path.
12. The roki daemon shall not accept a `[judge].model` configuration key, an `extension.linear_updater.*` namespace, or any other configuration surface tied to the removed setup-judge or linear-updater subprocess shapes; the loader shall log an actionable error if any such key is observed in the configuration source so that operators upgrading from a prior architecture can migrate.
13. The canonical configuration key reference is [`docs/reference/config.md`](../../../docs/reference/config.md); this requirement document does not restate per-key validation rules already documented there.
14. The roki daemon shall recognize the fixed Linear label names `roki:ready` and `roki:impl` (not operator-configurable). The pre-admission-judge ([fr:04-state-machine-and-recovery §Pre-admission judge](../../../docs/fr/04-state-machine-and-recovery.md)) shall evaluate every incoming Linear webhook against four ordered conditions: (a) `ticket.assignee == [linear].assignee`, (b) `ticket.linear_state ∈ [linear].admit_states`, (c) `roki:ready ∈ ticket.labels`, (d) presence of `roki:impl` to select the per-ticket `mode` (`SPEC_DRIVEN` if `roki:impl` is present alongside `roki:ready`; `NEEDS_CLASSIFY` if only `roki:ready` is present; `roki:impl` alone without `roki:ready` is treated as not authorized). On any failed condition the daemon shall silently skip the ticket (log only, no state entry, no Linear write, no orchestrator launch). On all conditions passing the daemon shall enter `Pending` with the chosen `mode` attached to the per-issue state.
15. The workspace-level `WORKFLOW.md` shall declare the four required named template blocks `prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, and `prompt_template_open_pr`. The first drives the orchestrator session's system prompt (with the per-ticket `mode` flag substituted in at render time). The remaining three drive the daemon-internal phase prompts for, respectively, the NEEDS_CLASSIFY (Path B / direct) `implement` phase, the NEEDS_CLASSIFY (Path B / direct) `validate` phase, and the `open_pr` phase (both modes). Operators may override any of these via the per-phase override surface in Requirement 6.7. If any of the four required blocks is missing at startup, the daemon shall refuse to start; if a hot reload removes one, the daemon shall retain the previous policy.

### Requirement 3: Linear Tracker Integration

**Objective:** As an operator, I want roki to discover and track only Linear issues that pass the pre-admission-judge with low overhead and respect Linear's rate limits, so that the daemon stays responsive without exhausting the API quota or handling tickets I have not explicitly authorized.

#### Acceptance Criteria
1. When a Linear webhook payload arrives at the single workspace-level webhook endpoint, the roki daemon shall verify the HMAC signature against the workspace-level webhook secret, normalize the payload into the internal issue model, and run the pre-admission-judge (Requirement 2.14) before updating the orchestrator's in-memory state.
2. If a webhook payload arrives without a valid signature header or with a signature that does not match the workspace-level webhook secret, the roki daemon shall reject the request with the documented unauthorized status code without normalizing the payload.
3. While webhook delivery is unavailable, the roki daemon shall poll Linear for issues whose Linear workflow state is in the configured `admit_states` set and whose assignee matches the configured Linear assignee, on a single workspace-level cadence that does not exceed once every five minutes. The pre-admission-judge runs on each polled issue identically to webhook-delivered ones.
4. If Linear returns an HTTP 429 response, the roki daemon shall apply exponential backoff before its next request and shall log the backoff window.
5. The roki daemon shall expose Linear data to the rest of the system only through a normalized issue model that includes at minimum the issue identifier, title, description, current state, label set, and assignee user identifier.
6. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from the orchestrator session (through the operator-configured Linear MCP integration) or from a phase subprocess (through the same MCP, where its `allowed_tools` permits it), and the daemon shall not register, proxy, or wrap any Linear write path on the agent's behalf.
7. When a webhook payload or poll result contains an issue that passes all four pre-admission-judge conditions (Requirement 2.14), the roki daemon shall admit the issue to `Pending` with the chosen `mode` and update the per-issue state machine.
8. If a webhook payload or poll result contains an issue that fails any pre-admission-judge condition, the roki daemon shall ignore the issue, shall not create a session tempdir, shall not launch the orchestrator session or any phase subprocess, shall not write to Linear, and shall emit a structured log event naming the issue identifier and the failed condition.
9. When a previously skipped issue subsequently passes all four pre-admission-judge conditions (typically because the operator added the `roki:ready` or `roki:impl` label), the roki daemon shall admit that issue on the next valid webhook or poll observation.
10. When an already-running issue is reassigned away from the configured Linear assignee, or has its `roki:ready` label removed mid-flight, the roki daemon shall stop further phase nominations for that issue, terminate any active phase subprocess and orchestrator session for the issue, route the issue into `Cleaning` without retry, and log the cause. The current orchestrator session is not retroactively re-routed if the `roki:impl` label is added or removed mid-flight (mode mutation mid-flight is out of scope per Requirement 4.10); the next webhook re-runs the pre-admission-judge.
11. The roki daemon shall maintain an in-memory deduplication index keyed by Linear issue identifier that records, for every currently-tracked issue, the issue's current daemon state, the per-issue `mode` flag (when in any non-terminal state), the most recently observed Linear state snapshot, and any in-flight subprocess handle (orchestrator session and phase subprocess).
12. When a webhook payload or poll result names an issue that already has an entry in the deduplication index in any non-terminal state (`Pending`, `Active`, `Backoff`, or `Cleaning`), the roki daemon shall not launch an additional orchestrator session or phase subprocess for that issue; the daemon shall update the recorded Linear state snapshot in place, and shall react to the new snapshot only through the explicit transitions documented in Requirements 3.10 and 4.6 (assignment loss, label removal, terminal Linear state).
13. When startup recovery and a runtime webhook concurrently observe the same issue, the roki daemon shall serialize their effects so that at most one orchestrator session and at most one phase subprocess are in flight per Linear issue identifier at any instant.
14. When a webhook payload or poll result names an issue whose deduplication entry is in the `Inactive` state and the new observation passes the pre-admission-judge, the roki daemon shall re-admit the issue by clearing the `Inactive` entry and starting a fresh admission cycle from `Pending` with the `mode` recomputed from the current Linear label set.

### Requirement 4: Per-Issue Session, Mode, and Worktree Lifecycle

**Objective:** As an operator, I want each pre-admission-passing Linear issue to be handed to the orchestrator with the correct per-ticket `mode` flag (SPEC_DRIVEN or NEEDS_CLASSIFY), and I want the daemon — not the agent — to materialize the worktree on every non-classify phase nomination (idempotent ensure: create on the first call, verify on subsequent calls) and reclaim it on cleanup, so that the orchestrator walks into a prepared workspace and the operator's allowlist remains the only enforcement boundary for git side-effects. Substantive admission classification (Path A/B/C/D/E) is performed inside the `classify` phase subprocess in NEEDS_CLASSIFY mode and inside the orchestrator's first deliberation turn against the operator-named target spec in SPEC_DRIVEN mode, per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md) and [fr:18-worker-skill-workflow](../../../docs/fr/18-worker-skill-workflow.md).

#### Acceptance Criteria
1. When the pre-admission-judge passes (transitioning the issue to `Pending`), the roki daemon shall launch the orchestrator session for the issue per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md), with the per-ticket `mode` flag rendered into the orchestrator's `prompt_template_orchestrator` system prompt at launch. The daemon shall not launch any separate setup-judge subprocess and shall not send any `admission_request` event — the orchestrator's first-turn behavior is determined by the rendered `mode`.
2. While the orchestrator session is running, the roki daemon shall write JSON events to the orchestrator's stdin per the event catalog in [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md) (`phase_complete`, `phase_nonclean`, `daemon_directive`, `tracker_terminal`) and shall parse the orchestrator's stdout one turn at a time. The daemon shall extract the **last** JSON object on the orchestrator's stdout per turn (after any extended-thinking block) and shall ignore reasoning text. The daemon shall validate the parsed object against the orchestrator's response schema (see [fr:19-orchestrator-session > Response schema](../../../docs/fr/19-orchestrator-session.md)), including the `action` and `phase` enums and the `outcome` enum on `action=stop`.
3. In SPEC_DRIVEN mode the orchestrator's first deliberation turn resolves the target spec name from the ticket body and structurally validates `<repo>/.kiro/specs/<target>/{spec.json,requirements.md,design.md,tasks.md}` (presence, non-emptiness, EARS keyword in `requirements.md`, actionable sub-task in `tasks.md`, `approvals.tasks.approved == true` in `spec.json`). On pass the orchestrator returns `action=run_phase phase=implement` with `additional_context` carrying the resolved target spec name and project-level spec dir path. On fail (target unresolvable, files missing, approvals incomplete) the orchestrator writes a Linear comment naming the missing artifact and the recommended `/kiro-spec-*` command via Linear MCP in the same turn, then returns `action=stop outcome=spec_incomplete`. The daemon routes the issue to `Inactive(reason=spec_incomplete)`.
4. In NEEDS_CLASSIFY mode the orchestrator's first turn returns `action=run_phase phase=classify`. On the resulting `phase_complete(classify)` event the orchestrator branches on `result.path`: Path B → `action=run_phase phase=implement` (direct mode); Path A / C / D / E → write a Linear comment quoting `result.suggested_command` and `result.suggested_label` via Linear MCP in the same turn, then `action=stop outcome=needs_operator`. The daemon routes the issue to `Inactive(reason=needs_operator)`.
5. The roki daemon shall validate every `action=stop` whose `outcome` names a target repo (when applicable) against the configured allowlist: out-of-allowlist or multi-repo target detected by the orchestrator surfaces as `outcome ∈ {needs_split, allowlist_rejected}` with an orchestrator-driven Linear write in the same turn; the daemon routes to `Inactive(reason=needs_split)` or `Inactive(reason=allowlist_rejected)` respectively.
6. When the orchestrator nominates any phase **other than `classify`** for a validated single allowlisted repository, the roki daemon shall ensure a git worktree exists for the issue before spawning the phase subprocess. The worktree-setup operation is **idempotent**: on the first such nomination the daemon shall resolve the repository's local checkout via `ghq list -p` and create a single git worktree via `wt switch-create` whose branch name equals the Linear issue identifier verbatim; on every subsequent non-`classify` nomination the daemon shall verify the worktree's continued presence via `wt list` (or equivalent) without invoking `wt switch-create` again. The daemon shall also ensure the per-issue session tempdir exists under the platform-appropriate user cache root using the Linear issue identifier as the directory name (created on entry to `Pending`, idempotent on subsequent verification). The resulting worktree path is exposed to every phase subprocess invocation that needs it (every phase except `classify`) through its per-phase context envelope. The `classify` phase MUST NOT receive a worktree path, since `classify` runs against ticket context and the project-level spec dir alone, before any allowlisted repo has been resolved.
7. If the orchestrator's response fails JSON schema validation, the roki daemon shall reprompt the orchestrator once per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md); on a second consecutive schema-drift the daemon shall route the issue to `Inactive(reason=orchestrator_unparseable)`.
8. The roki daemon shall reject any session or worktree path that, after sanitization, escapes its expected root, contains path traversal segments, or collides with another active issue's session.
9. The roki daemon shall enter the `Cleaning` state for an issue only when the tracker observes (via webhook or poll) one of: (a) the Linear issue transitioned to a Linear state configured as terminal (e.g., done, canceled), or (b) the Linear issue was reassigned away from the configured assignee, or (c) the `roki:ready` label was removed. If the orchestrator session or any phase subprocess is still running for that issue at the moment of observation, the daemon shall terminate them before performing cleanup. While in `Cleaning`, the daemon shall locate the worktree whose branch name equals the Linear issue identifier verbatim under the configured allowlisted repository's local checkout, remove it via `wt remove`, and remove the session tempdir; the daemon shall not delete any branches.
10. The per-ticket `mode` flag is **immutable** for the lifetime of the orchestrator session. Relabeling the ticket while the orchestrator is running does not re-route an in-flight ticket; the next webhook re-runs the pre-admission-judge per Requirement 3.14.
11. When an issue lands in any `Inactive(reason=*)` failure state (any non-`awaiting_linear` reason), the roki daemon shall retain the worktree, the branch, and the session tempdir so that the operator can inspect the residue, and shall not auto-clean until the tracker observes the issue's Linear state moving to a configured terminal state. `Inactive(reason ∈ {needs_operator, spec_incomplete})` are explicitly not auto-cleanup eligible.
12. If session creation, worktree creation, or worktree removal fails, the roki daemon shall mark the corresponding issue as `Inactive(reason=fs_poison)`, log the filesystem error with the offending path, send a `daemon_directive` event with kind `fs_poison` to the orchestrator (when the orchestrator is still alive) so that the orchestrator surfaces the failure to Linear, and refuse to start additional work for that issue until the operator intervenes. If the orchestrator is no longer alive at the moment of `fs_poison`, the failure surfaces via structured log + TUI escalation queue only, per Requirement 12.

### Requirement 5: Bounded Subprocess Adapters (Orchestrator Session and Phase Subprocesses)

**Objective:** As an operator, I want each admitted issue to be driven by a long-lived orchestrator session whose JSON action directives nominate short-lived bounded phase subprocesses for the actual code-changing work, so that the daemon supervises both shapes as a thin observation layer with bounded thinking effort (`max_phases`) on the orchestrator and bounded turn budgets (`--max-turns`) on each phase, without driving any agent loop itself.

The two engine shapes the daemon supervises per ticket are: (a) **orchestrator session** — one long-lived `claude --input-format stream-json --output-format stream-json` per ticket, launched on entry to `Pending`, canonicalized in [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md); and (b) **phase subprocesses** — zero or more short-lived subprocesses (skill-driven `claude -p '/<skill> <args>' --output-format stream-json` for `classify` / SPEC_DRIVEN `implement` / `validate` / `review` / `ci_fix` / `finalize_review`, or template-driven `claude --input-format stream-json --output-format stream-json` for NEEDS_CLASSIFY `implement` / `validate` and `open_pr`) that the orchestrator nominates via `action=run_phase`. The setup-judge, linear-updater, and previous per-issue `materialize_spec` subprocess shapes from the prior architecture are removed; the prior daemon-side mechanical kiro-spec / kiro-review gates are also removed (their work is absorbed by the orchestrator as artifact validation per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md)).

#### Acceptance Criteria
1. When the pre-admission-judge passes (`Discovered → Pending`), the roki daemon shall launch the orchestrator session per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md): one `claude --input-format stream-json --output-format stream-json` subprocess in the issue's session tempdir, with `--settings` enforcing the `extension.orchestrator.allowed_tools` allowlist (Linear MCP write + `Read` + `Bash` by default), with the `prompt_template_orchestrator` block rendered as system prompt with the per-ticket `mode` flag substituted in, and with a read-only filesystem sandbox (Bash invocations execute inside the sandbox so they cannot mutate the worktree).
2. While the orchestrator session is running, the roki daemon shall write JSON events to the orchestrator's stdin per the event catalog in [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md) and shall parse the orchestrator's stdout one turn at a time. The daemon shall extract the **last** JSON object on the orchestrator's stdout per turn (after any extended-thinking block) and shall ignore reasoning text. The daemon shall validate the parsed object against the orchestrator's response schema (see [fr:19-orchestrator-session > Response schema](../../../docs/fr/19-orchestrator-session.md)), including the `action`, `phase`, and `outcome` enums.
3. If the orchestrator emits no stdout for longer than the configured `extension.orchestrator.stall_seconds` window (default `120`), the roki daemon shall SIGTERM the orchestrator and route the issue to `Inactive(reason=orchestrator_crash)` per Requirement 12. The daemon shall not write a Linear-side notification on the orchestrator's behalf in this case.
4. If the orchestrator's response fails JSON schema validation on two consecutive turns (after one daemon-side reprompt), the roki daemon shall route the issue to `Inactive(reason=orchestrator_unparseable)` per Requirement 12 and shall capture the orchestrator's raw stdout in the structured log.
5. The roki daemon shall enforce the orchestrator's `max_phases` budget: each `action=run_phase` directive the orchestrator emits consumes one slot of `extension.orchestrator.max_phases` (default `15`); when the budget is exhausted and the orchestrator would nominate another phase, the daemon shall route the issue to `Inactive(reason=orchestrator_budget_exhausted)` per Requirement 12 and shall not spawn the additional phase. There is no per-turn budget on the orchestrator; bounding is via `max_phases` and `effort`.
6. When the orchestrator emits `action=run_phase` with `phase ∈ {classify, implement, review, validate, open_pr, ci_fix, finalize_review}` and `additional_context` (when present), the roki daemon shall spawn the corresponding phase subprocess. The catalog default invocation per phase is documented in [fr:18-worker-skill-workflow](../../../docs/fr/18-worker-skill-workflow.md) and is mode-aware where it matters: `classify` (NEEDS_CLASSIFY only, first turn only; default skill `roki-classify`, max_turns 5); `implement` (SPEC_DRIVEN default skill `kiro-impl <target>`, NEEDS_CLASSIFY default daemon-internal `prompt_template_implement_direct`); `review` (default skill `kiro-review` with mode-dependent criteria source); `validate` (SPEC_DRIVEN default skill `kiro-validate-impl <target>`, NEEDS_CLASSIFY default daemon-internal `prompt_template_validate_direct`); `open_pr` (daemon-internal `prompt_template_open_pr`, no default skill); `ci_fix` (default skill `roki-ci-fix`); `finalize_review` (default skill `roki-finalize-review` with mode-dependent criterion ID source). When the workspace-level `WORKFLOW.md` declares a per-phase override (per Requirement 6.7), the daemon shall apply that override to the corresponding phase's invocation. The daemon shall pass the configured per-phase `--max-turns` budget, render the per-phase context envelope (including the orchestrator's `additional_context` verbatim through the engine adapter's `additional_context` channel per Requirement 13.4), and stream the phase subprocess's stdout as newline-delimited JSON events parsed into typed lifecycle events for structured logging.
7. If a phase subprocess emits no events for longer than the configured per-phase `extension.phase.<name>.stall_seconds` window (default `120` for every phase; per-phase override allowed), the roki daemon shall terminate the subprocess, send a `phase_nonclean` event to the orchestrator with the stall classification, and let the orchestrator decide whether to re-run the phase, fall through to a `ci_fix` phase, or `action=stop`. If the orchestrator is no longer alive when the phase stall is detected, the daemon shall route the issue to `Inactive(reason=stall)` and surface the failure via structured log + TUI escalation queue only.
8. When a phase subprocess clean-exits with terminal `result.subtype = success`, the roki daemon shall send a `phase_complete` event to the orchestrator with the structured exit envelope (including `result.path` for `classify`, `pr_url` for `open_pr`, `review_artifact_path` for `finalize_review`) and shall await the orchestrator's next directive. When the phase clean-exits with a non-`success` `result.subtype`, exits non-zero without a terminal `result`, is signal-terminated, or exhausts its `--max-turns`, the daemon shall send a `phase_nonclean` event to the orchestrator with the structured failure classification and shall await the orchestrator's next directive. **Exception**: when the phase subprocess exit was caused by a daemon-issued SIGTERM in response to a tracker terminal observation (Linear state moved to `done` / `canceled`, assignee reassigned, or `roki:ready` removed), the daemon shall NOT translate the exit into `phase_complete` / `phase_nonclean`; the exit event shall be discarded (captured in the structured log only) and the daemon shall deliver `tracker_terminal` solo so the orchestrator's next turn returns `action=stop outcome=cancelled` without deliberating an already-overridden phase outcome.
9. When the terminal `result` event reports a `subtype` not present in the daemon's compiled mapping — including any new `subtype` Claude Code adds after the daemon's build — the roki daemon shall include the raw `subtype` value in the `phase_nonclean` event payload to the orchestrator and capture it in the structured log; the daemon does not unilaterally route to `Inactive` for an unknown phase `subtype` because the orchestrator is alive to make that judgment.
10. When the configured ticket-level retry budget for phase non-clean exits is exhausted (default three attempts, range one through ten), the roki daemon shall send a `daemon_directive` event with kind `retry_exhausted` to the orchestrator so that the orchestrator surfaces the failure to Linear and emits its own `action=stop` with `outcome=failure`. While remaining attempts exist the daemon shall transition the issue from `Active` to `Backoff`, apply exponential backoff between attempts bounded between ten seconds and five minutes, retain the session tempdir and the worktree across retries, and on timer expiry transition back to `Active` for the orchestrator's next phase nomination.
11. When the orchestrator emits `action=stop`, the roki daemon shall record the terminal `outcome` value (one of `success`, `failure`, `cancelled`, `needs_operator`, `spec_incomplete`, `needs_split`, `allowlist_rejected`), publish the `Pending → Inactive` transition (no subscriber veto — the prior review-gate vetoable hook is removed; structural validation of `review.md` is owned by the orchestrator itself per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md)), and gracefully terminate the orchestrator per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md). The mapping from `outcome` to `Inactive.reason` is documented in [fr:04-state-machine-and-recovery](../../../docs/fr/04-state-machine-and-recovery.md).
12. The `classify` phase (NEEDS_CLASSIFY mode, first turn only) is invoked with `--max-turns 5` and runs `roki-classify` (a daemon-purpose-built skill at `.claude/skills/roki-classify/SKILL.md` whose tool surface is `Read` + `Glob` + `Grep` only, with no dialogue and no file writes). Its terminal `result` event payload shall include `path ∈ {A, B, C, D, E}`, `target_feature` (when `path == A`), `suggested_command`, `suggested_label`, and `rationale`. The orchestrator branches on `path` as documented in Requirement 4.4.

### Requirement 6: Workspace-Level WORKFLOW.md Policy Loader

**Objective:** As an operator, I want a single workspace-level `WORKFLOW.md` file that defines orchestrator-session and phase policy in Liquid + Markdown with schema validation and hot reload, so that I can adjust orchestrator and phase behavior without restarting the daemon or recompiling Rust.

#### Acceptance Criteria
1. When the roki daemon starts, the WORKFLOW.md loader shall read the configured workspace-level `WORKFLOW.md`, parse its YAML or TOML front matter, identify the four required named template blocks (`prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, `prompt_template_open_pr`) plus zero or more optional `prompt_template_<phase>` named template blocks (per-phase override surface for phase subprocesses, per Requirement 6.7), and validate the result against the published schema. The previous `prompt_template_setup`, `prompt_template_worker`, and `prompt_template_linear_updater` named blocks are removed alongside the subprocess shapes they served.
2. If the workspace-level `WORKFLOW.md` fails schema validation at startup, the roki daemon shall refuse to start and shall log the validation error with the offending key path.
3. While the daemon is running, the WORKFLOW.md loader shall watch the workspace-level `WORKFLOW.md` for filesystem changes and shall re-validate the file before applying any changes. Hot-reload of the orchestrator namespace and the per-phase template blocks applies from the next ticket admission; an in-flight orchestrator keeps its rendered prompt and an in-flight phase keeps its rendered prompt for the lifetime of the session.
4. If a hot-reload attempt produces a `WORKFLOW.md` that fails validation, the roki daemon shall keep the previously valid policy in effect and log the validation failure.
5. The WORKFLOW.md loader shall expose its parsed policy to the orchestrator and the phase subprocesses through a stable schema shape that downstream specs can extend with additional keys without breaking existing consumers, and the loader shall round-trip unknown keys under the reserved extension namespaces (`extension.orchestrator.*`, `extension.phase.<name>.*`, `extension.server.*`) without interpreting them.
6. When the roki daemon launches the orchestrator session, the WORKFLOW.md loader shall render the `prompt_template_orchestrator` block against the active Linear issue with the issue's identifier, title, description, label list, the per-ticket `mode` flag, and bucketed lifecycle state available as named variables; the rendered output shall become the orchestrator's system prompt input. If rendering fails, the daemon shall log the render failure with the offending block name and shall provide a deterministic fallback prompt that includes the Linear issue identifier, title, description, and `mode` flag so that the orchestrator always receives non-empty issue context.
7. Each phase ships a daemon-built default invocation (a slash-command-driven skill in the operator's installed `~/.claude/skills/<skill>` for `classify` / SPEC_DRIVEN `implement` / `review` / SPEC_DRIVEN `validate` / `ci_fix` / `finalize_review`, or a daemon-internal Liquid template for NEEDS_CLASSIFY `implement` / `validate` and `open_pr`) per [fr:18-worker-skill-workflow](../../../docs/fr/18-worker-skill-workflow.md). Operators MAY override per phase via the workspace-level `WORKFLOW.md` using either an `extension.phase.<name>.command` key (slash-command swap; the daemon launches `claude -p '<command>' --output-format stream-json --max-turns N`) or a named `prompt_template_<phase>` template block (Liquid + Markdown rendered against the per-phase variables and written to the subprocess's stdin; the daemon launches `claude --input-format stream-json --output-format stream-json --max-turns N`). The two override forms (`command` xor `prompt_template_<phase>`) are mutually exclusive per phase: if the workspace-level `WORKFLOW.md` declares both for the same phase, the roki daemon shall refuse to start at startup or retain the previous policy at hot reload, and shall log the offending phase name. Independent of the prompt-override form, operators MAY also set `extension.phase.<name>.max_turns` to override the catalog `--max-turns` default and `extension.phase.<name>.stall_seconds` to override the per-phase stall window default (Requirement 5.7). These two scalar keys are additive — they may be set with or without `command`, and they coexist with `prompt_template_<phase>`. Absent either prompt-override form for a given phase, the daemon shall use the catalog default for the active mode. Hot reload of an override shall apply from the next nomination of that phase; an in-flight phase subprocess always finishes with the policy in effect at spawn. The daemon shall pass any per-phase `additional_context` from the orchestrator through the engine adapter's stable forwarding channel (Requirement 13.4) regardless of which form is used. Phase override `<name>` values: `classify`, `implement`, `review`, `validate`, `open_pr`, `ci_fix`, `finalize_review`.

### Requirement 7: Agent Tooling Boundary

**Objective:** As an operator, I want the daemon to neither register nor proxy any agent-side tool, so that every agent invocation (orchestrator session, every phase subprocess) inherits exactly the tool surface my local Claude Code installation exposes (Bash plus my installed MCP servers), narrowed only by per-process `allowed_tools`, and the daemon never needs to hold Linear or git write credentials on the agent's behalf.

#### Acceptance Criteria
1. The roki daemon shall not register, proxy, or wrap any agent-side tool inside the orchestrator session or any phase subprocess; each subprocess's tool surface shall be exactly what the operator's local Claude Code installation exposes — its built-in tool set plus any MCP servers the operator has installed — narrowed only by per-process `allowed_tools` lists the daemon assembles from configuration. For the orchestrator specifically the surface is Linear MCP (write) + `Read` + `Bash` (read-only sandbox) per [fr:19-orchestrator-session > Tool surface](../../../docs/fr/19-orchestrator-session.md). For the `classify` phase specifically the surface is `Read` + `Glob` + `Grep` only per [fr:11-agent-tool-boundary](../../../docs/fr/11-agent-tool-boundary.md).
2. The roki daemon shall never issue Linear write operations from within its own process; all Linear writes shall originate from the orchestrator session (through the operator-configured Linear MCP integration) or from a phase subprocess where its `allowed_tools` permits, and the daemon shall not hold any agent-facing Linear write path.
3. Where the workspace-level `WORKFLOW.md`, the orchestrator's rendered prompt, or a phase subprocess's installed skill prompt instructs an agent invocation to operate `git`, `gh`, `ghq`, `wt`, or Linear tooling from inside the subprocess, the operator's local Claude Code installation shall be the source of those affordances (Bash for shell tools and the operator's installed MCP servers for Linear); the roki daemon shall not substitute, intercept, or augment those tool invocations.
4. The roki daemon shall never embed the Linear API token or the Linear webhook secret in any prompt input, log line, or other artifact reachable from inside any agent subprocess.

### Requirement 8: Orchestrator State Machine and Extension Points

**Objective:** As a downstream spec author, I want the orchestrator to expose stable per-issue state-machine hooks, so that roki-observability can subscribe to lifecycle transitions without forking the core.

#### Acceptance Criteria
1. The roki daemon shall maintain an in-memory state machine per Linear issue identifier covering exactly the following five daemon-local states: `Pending`, `Active`, `Backoff`, `Inactive`, and `Cleaning`. The previous `Judging` state is removed alongside the `admission_request` / `admission_decision` round-trip. These states represent the daemon's bookkeeping for subprocess lifecycle and cleanup; the daemon does not mirror Linear's workflow states (review, done, etc.). `Inactive` shall carry a `reason` discriminator drawn from the set `{awaiting_linear, needs_operator, spec_incomplete, needs_split, allowlist_rejected, orchestrator_crash, orchestrator_unparseable, orchestrator_budget_exhausted, stall, retry_exhausted, fs_poison, orphan}`, used for structured logging, TUI surfacing, and `Cleaning` eligibility but not for orchestrator-internal transition decisions. `Pending` is the staging state covering admission-passed-orchestrator-launched and any non-running idle waiting between phases (it is the entry state from a pre-admission-judge pass and from re-admission of an `Inactive` entry per Requirement 3.14, and the daemon also returns to `Pending` after each `Active` clean exit while the orchestrator deliberates the next directive). The orchestrator session is launched on entry to `Pending` per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md) with the per-ticket `mode` flag rendered into its system prompt. `Backoff` is the retry-window state for ticket-level retries between phase subprocess non-clean exits; on timer expiry it returns to `Active` for the orchestrator's next phase nomination. `Cleaning` shall be reachable only from `Pending`, `Active`, `Backoff`, or `Inactive`, and only as a result of a tracker observation (the issue moved to a configured terminal Linear state, the issue was reassigned, or the `roki:ready` label was removed); a phase subprocess exit alone shall never trigger entry to `Cleaning`. The state machine shall declare no vetoable transitions: the prior `Judging → Active` (spec gate) and `Active → Inactive` (review gate) hooks are removed alongside the gates and the `Judging` state itself; structural validation of `review.md` and the SPEC_DRIVEN target spec docs is owned by the orchestrator inside its own phase-planning loop per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md).
2. When a state transition occurs, the roki daemon shall publish a structured transition event whose payload identifies the previous state, the next state, the trigger source, and the issue identifier, and shall include the originating repository identifier when one applies. When the next state is `Inactive`, the payload shall additionally include the `reason` discriminator. When the next state is `Pending` from an admission entry, the payload shall additionally include the per-ticket `mode` flag.
3. The roki daemon shall expose subscription hooks that allow other components to observe transition events read-only; no transition is vetoable.
4. If a subscriber raises an unhandled error while processing a transition event, the roki daemon shall isolate the failure to that subscriber, log the error with the subscriber's identifier, and continue processing transitions for other subscribers.
5. The roki daemon shall recover the per-issue state on restart by re-reading Linear and the filesystem layout, without depending on a persistent database. The orchestrator session is not persisted across daemon restarts: a fresh orchestrator is launched when an issue re-enters `Pending` per [fr:19-orchestrator-session > Lifecycle](../../../docs/fr/19-orchestrator-session.md), with the `mode` flag recomputed from the current Linear label set.

### Requirement 9: Permission Strategy and Default Sandbox

**Objective:** As an operator, I want roki to run agents under the safest workable permission profile while letting me opt into a fallback when Claude's allowlist is unreliable, so that I can balance safety against operability.

#### Acceptance Criteria
1. The roki daemon shall launch each phase subprocess with the agent sandbox set to `workspace-write` and with elicitations rejected by default.
2. When the workspace-level `WORKFLOW.md` declares an alternative sandbox or elicitation policy, the roki daemon shall apply that override to every phase subprocess.
3. When the operator selects the `--settings` allowlist permission strategy, the roki daemon shall pass the configured allowlist to each phase subprocess through Claude Code's settings interface.
4. When the operator selects the `--dangerously-skip-permissions` fallback strategy through configuration or the `--dangerously-skip-permissions` CLI flag, the roki daemon shall pass that flag to each phase subprocess and shall log the elevated-permission decision per phase launch. The fallback strategy shall not apply to the orchestrator session; the orchestrator's tool surface is governed exclusively by `extension.orchestrator.allowed_tools` via `--settings`.
5. If neither permission strategy is configured for phase subprocesses, the roki daemon shall refuse to start and shall report the missing configuration.
6. The roki daemon shall launch the orchestrator session with a read-only filesystem sandbox and rejected elicitations regardless of operator overrides per [fr:19-orchestrator-session > Tool surface](../../../docs/fr/19-orchestrator-session.md). The orchestrator's `Bash` capability (when permitted by `extension.orchestrator.allowed_tools`) executes inside that sandbox so commands can read state but cannot mutate the worktree or session tempdir. Whether the orchestrator can call write-capable Linear MCP tools is governed by the operator's Claude Code MCP allowlist intersected with `extension.orchestrator.allowed_tools`, not by the daemon's permission strategy.

### Requirement 10: Restart Recovery Without Persistent Storage

**Objective:** As an operator, I want roki to recover its working state after a restart by re-reading Linear and the filesystem rather than relying on a database, so that I never carry stale or corrupted local state across restarts.

#### Acceptance Criteria
1. When the roki daemon starts, it shall list every session tempdir under the platform-appropriate user cache root and, for every configured allowlisted repository, list every existing git worktree whose branch name matches the operator-configurable Linear issue-identifier pattern.
2. The roki daemon shall reconcile every distinct issue identifier discovered from session tempdirs and from worktree branches against Linear before resuming work, shall apply the pre-admission-judge during reconciliation (assignee + Linear state + label set), and shall classify each discovered issue into one of the documented recovery states (resume-active, orphaned-session, orphaned-worktree, fresh-queued, no-op), mapping resume-active to `Pending` (which then re-launches a fresh orchestrator session with the `mode` flag recomputed from the current Linear label set) and the orphan classifications to `Inactive(reason=orphan)`.
3. If a session tempdir or worktree exists for an issue identifier that has no matching pre-admission-passing Linear issue at startup, the roki daemon shall mark that residue as `Inactive(reason=orphan)`, log it without deleting it automatically, and — if and when a future orchestrator session is launched for any in-flight issue — surface the orphan via a `daemon_directive` event with kind `orphan` so that the orchestrator writes the appropriate Linear label + comment. If no in-flight orchestrator exists, the orphan surfaces via structured log + TUI escalation queue only, per Requirement 12.
4. If a Linear issue passes the pre-admission-judge at startup but has no matching session tempdir or worktree, the roki daemon shall enqueue the issue for a fresh admission cycle starting from `Pending` (which launches a fresh orchestrator session with the chosen `mode`).
5. The roki daemon shall not write any per-issue runtime state to disk except the session tempdir and worktree contents the agent itself produces and the structured logs the daemon emits.

### Requirement 11: Observability of Daemon Internals

**Objective:** As an operator debugging a stuck ticket, I want enough structured observability from the daemon itself to diagnose orchestrator, phase, session, worktree, and tracker issues without an external UI, so that the MVP is operable before the dedicated observability spec ships.

#### Acceptance Criteria
1. The roki daemon shall emit a structured log event for every pre-admission-judge evaluation (including skip outcomes with the failed condition), every orchestrator session lifecycle change, every phase subprocess lifecycle change, every session tempdir creation or deletion, every worktree creation or removal, every Linear poll or webhook receipt, every backoff or stall decision, every retry attempt with its attempt counter, every `daemon_directive` event sent to the orchestrator, every orchestrator response (action + phase + outcome fields, redacted of `reason` when long), and every state-machine transition (including the `Inactive` `reason` discriminator and the per-ticket `mode` on admission). The canonical event catalog lives in [`docs/reference/log-events.md`](../../../docs/reference/log-events.md).
2. The roki daemon shall include the Linear issue identifier when one applies, the repository identifier when one applies (for example for worktree-scoped events), and a correlation identifier for the originating subprocess invocation in every structured log event.
3. The roki daemon shall support a configurable log level and a configurable log destination so that the operator can route logs to stdout, a file, or both.
4. The roki daemon shall redact the Linear API token, the Linear webhook secret, and any other operator-declared secrets from every structured log event before emission.
5. While the orchestrator session or any phase subprocess is running, the roki daemon shall capture every line emitted on the subprocess's standard error stream and shall surface each non-empty line as a structured log event at warn severity tagged with the issue identifier, the subprocess role (`orchestrator` or `phase:<phase-name>`), and the correlation identifier.
6. Where the operator opts into per-issue debug capture through the documented CLI flag or configuration block, the roki daemon shall append every line emitted on each subprocess's standard output and standard error streams to a per-issue file under the configured debug-log directory, and each appended line shall carry an RFC 3339 timestamp with nanosecond resolution and a stream tag identifying standard output or standard error.
7. If the per-issue debug log file cannot be opened or appended to, the roki daemon shall log the failure with the offending file path and shall continue running the subprocess without aborting the launch.
8. When the orchestrator session or any phase subprocess completes (success, retry, or final failure), the roki daemon shall emit a structured log event recording the subprocess role, the duration, the parsed outcome (when parseable), and the issue identifier.

### Requirement 12: Daemon-Only Failure Surfacing

**Objective:** As an operator, I want roki to surface daemon-only failures that no agent has yet self-reported — into Linear (when the orchestrator session is alive to do it on the daemon's behalf) and into the optional TUI escalation queue (always for daemon-only failures; TUI-only for the three orchestrator-dead reasons) — so that I learn about stuck or failed tickets without continuously tailing daemon logs and without holding any Linear write path inside the daemon process.

The canonical surfacing channel is **the orchestrator processing a `daemon_directive` event and writing Linear via Linear MCP**. When the orchestrator is dead — because of `orchestrator_crash`, `orchestrator_unparseable`, or `orchestrator_budget_exhausted` — the daemon does **not** fall back to a Linear write of its own; the failure surfaces via structured log + TUI escalation queue only, per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md).

#### Acceptance Criteria
1. The roki daemon shall maintain an in-memory escalation queue keyed by Linear issue identifier that records the most recent daemon-only failure event per issue (failure category, structured fields, timestamp, correlation identifier when applicable, repository identifier when applicable). The escalation queue is the data source consumed by the optional `OrchestratorRead` snapshot exposed to roki-observability.
2. When the daemon detects a daemon-only failure event listed below **and the orchestrator session is alive for the affected issue**, the roki daemon shall send a `daemon_directive` event to the orchestrator's stdin per [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md). The orchestrator then writes the appropriate Linear label + comment via Linear MCP and returns `action=linear_update_done`. The `daemon_directive` covers:
   (a) phase stall after stall detection (Requirement 5.7) — the daemon killed the phase subprocess so the phase agent had no chance to self-report;
   (b) retry budget exhaustion after phase non-clean exits (Requirement 5.10);
   (c) filesystem poison during session or worktree creation, removal, or rename (Requirement 4.12);
   (d) startup-recovery orphan retained for inspection (Requirement 10.3);
   (e) any other daemon-detected failure where the orchestrator is alive and Linear-side surfacing is needed.
   The daemon does **not** send a `daemon_directive` for operator-facing pre-phase stops (`outcome ∈ {needs_split, allowlist_rejected, spec_incomplete, needs_operator}`) — the orchestrator writes those Linear updates itself in the same turn it returns `action=stop`, per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md). `review.md` validation retry-budget exhaustion (after `finalize_review`) is owned by the orchestrator entirely — the orchestrator writes the Linear feedback itself and emits `action=stop` with `outcome=failure`; the daemon does not send a directive for it.
3. When the daemon detects a failure that kills the orchestrator itself, the roki daemon shall route the issue to one of three `Inactive(reason=*)` values per [fr:19-orchestrator-session > Failure modes](../../../docs/fr/19-orchestrator-session.md) and shall **not** attempt a Linear write on the orchestrator's behalf:
   (a) A process crash, SIGSEGV, non-zero exit without a `stop` action, or orchestrator stall (no event emitted in N seconds, configurable) → `Inactive(reason=orchestrator_crash)`;
   (b) A schema drift on two consecutive turns after one daemon-side reprompt → `Inactive(reason=orchestrator_unparseable)`;
   (c) `max_phases` exhausted (the orchestrator would nominate another phase but the budget is gone) → `Inactive(reason=orchestrator_budget_exhausted)`.
   In all three cases the daemon shall log the failure structurally, populate the escalation queue (Requirement 12.1), and surface the issue exclusively via the TUI escalation queue from roki-observability — there is no Linear-side notification. These three reasons are **not** auto-cleanup eligible: the worktree and session tempdir are preserved until the operator manually closes the Linear ticket, after which `Cleaning` proceeds.
4. The roki daemon shall not send `daemon_directive` events for events that an agent (the orchestrator or a phase subprocess) is expected to self-report through Linear, including normal phase completions, agent-recoverable errors the agent surfaces in the ticket itself, and the operator-facing pre-phase stops listed in Requirement 12.2.
5. Each `daemon_directive` event shall pass at minimum the Linear issue identifier, the daemon-only failure `kind`, the structured fields associated with the failure (e.g. `correlation_id`, `repos[]`, `worktree_path`, `last_subtype`, `attempts`, `window_ms`, `errno`), and the event timestamp; the directive shall not include the Linear API token, the Linear webhook secret, or any other operator-declared secret.
6. The actual Linear label name(s) and comment text the orchestrator applies are determined by the orchestrator's `prompt_template_orchestrator` system prompt and any operator instructions therein; the daemon contributes only the directive `kind` and the structured fields and does not template the Linear write itself.
7. If the orchestrator's response to a `daemon_directive` indicates a partial Linear write (the `linear_writes` field in `action=linear_update_done` shows a subset of the expected writes), the roki daemon shall log the partial-write entry, retain the escalation queue entry, and shall not retry on the orchestrator's behalf. If the orchestrator fails entirely on a `daemon_directive` (turn ends with an error, or the orchestrator crashes mid-turn), the daemon shall log the failure and route the issue to `Inactive(reason=orchestrator_crash)` per Requirement 12.3 — the escalation queue entry stays so the TUI continues to surface it.

### Requirement 13: Cross-Spec Extension Surface

**Objective:** As a downstream spec author, I want roki-mvp to publish a stable, additive extension surface, so that roki-observability can integrate without forking the core or breaking sibling specs.

#### Acceptance Criteria
1. The roki daemon shall publish a read-only `OrchestratorRead` trait that exposes a snapshot of the per-issue state (including the `Inactive` `reason` discriminator with the three orchestrator-dead values from Requirement 12.3 and the operator-facing values from Requirement 8.1) plus a single-issue lookup, plus a snapshot of the escalation queue from Requirement 12.1, and shall grant no state-mutation rights through that trait.
2. The roki daemon shall declare no vetoable state transitions; the prior `Judging → Active` (spec gate) and `Active → Inactive` (review gate) hooks are removed alongside the gates and the `Judging` state itself. Structural validation of `review.md` and the SPEC_DRIVEN target spec docs is owned by the orchestrator session inside its own phase-planning loop per [fr:19-orchestrator-session](../../../docs/fr/19-orchestrator-session.md), not by a daemon-side subscriber.
3. The roki tracker adapter shall publish a `TrackerRefresh` nudge trait that allows external callers to request an out-of-cycle poll without bypassing the documented cadence cap or the 429 backoff state.
4. The roki engine adapter shall accept an additive optional `additional_context` field on the per-phase context and shall forward its value verbatim to the phase subprocess through a stable, machine-extractable section of the phase subprocess's prompt envelope, kept distinct from the skill's installed prompt body, without the daemon interpreting the contents. The exact serialization shape (delimiter sentinels, JSON envelope, named keys) is defined by the engine adapter design and is treated as additive: new optional keys may be added under the same forwarding contract; removing or retyping existing keys is a breaking change. The orchestrator populates `additional_context` directly on `action=run_phase` directives — including the `review.md` validation retry path (failing per-criterion entries injected into the next `implement` phase) and the SPEC_DRIVEN first-`implement` nomination (target spec name + project-level spec dir path) — per [fr:19-orchestrator-session > Event catalog](../../../docs/fr/19-orchestrator-session.md).
5. The workspace-level `WORKFLOW.md` schema shall reserve the `extension.orchestrator.*`, `extension.phase.<name>.*`, and `extension.server.*` sub-namespaces for the canonical roki specs and the orchestrator session; the loader shall round-trip unknown keys under those namespaces without interpreting them. Per-phase named template blocks (`prompt_template_<phase>`) live alongside the four required template blocks (`prompt_template_orchestrator`, `prompt_template_implement_direct`, `prompt_template_validate_direct`, `prompt_template_open_pr`) as named template blocks (Requirement 6.1); they are part of the per-phase override surface defined in Requirement 6.7 and are mutually exclusive per phase with `extension.phase.<name>.command`. The previously reserved `extension.linear_updater.*`, `extension.gates.spec.*`, and `extension.gates.review.*` namespaces are removed alongside the subprocess shapes / gates they served.
